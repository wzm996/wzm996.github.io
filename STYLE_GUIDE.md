# 博客写作风格规范（STYLE_GUIDE）

> 目标：把 wzm996.github.io 写成「技术专栏型」博客：系列化、工程化、可复现。新文章看起来要像这个博客里原本就会出现的文章。

## 1. 文章模板与硬性格式

- 新文章必须使用 `scaffolds/post.md` 的 Front-matter 字段：
  - `title` / `date` / `categories`（层级）/ `tags`
- 正文必须在合适位置插入摘录分隔符：

```html
<!-- more -->
```

- 建议保持与历史文章一致的开头结构（强烈推荐）：
  1) Front-matter
  2) `<hr>`
  3) 一段引言/定义式摘要（使用 `blockquote`，1~3 句话）
  4) `<!-- more -->`

> 注：仓库历史文章目前常使用 HTML 标签（如 `<hr>`、`<blockquote><p>...</p></blockquote>`、`<h1>` 等）。为了“风格对齐”，新文章允许/鼓励继续沿用这种写法；若后续要迁移为纯 Markdown，再统一改风格。

## 2. 分类（categories）与标签（tags）

### 2.1 分类白名单（必须从中选，禁止自造新分类名/二级分类名）

- 数据结构与算法 / 复杂度与分析｜基础数据结构｜树与图｜排序与查找｜算法思想
- 计算机网络 / TCP/IP 基础｜HTTP/HTTPS｜DNS/CDN｜代理与负载均衡｜抓包与排障｜网络安全基础
- 操作系统与 Linux / 进程与线程｜内存管理｜文件系统｜I/O｜Linux 工具链｜性能分析
- 编程语言 / Java｜Go｜Python｜JavaScript/TypeScript｜Rust
- 后端工程 / Web 开发基础｜鉴权与安全｜任务与调度｜可观测性｜性能与高并发
- 数据库与存储 / MySQL｜Redis｜Elasticsearch｜消息队列｜事务与一致性
- 云原生与 DevOps / Docker｜Kubernetes｜CI/CD｜IaC｜发布与回滚
- 架构与设计 / 设计模式｜DDD｜微服务与治理｜缓存与一致性｜系统设计

### 2.2 categories 的写法（Hexo 层级）

示例：

```yaml
categories:
  - 计算机网络
  - HTTP/HTTPS
```

### 2.3 tags 的写法

- 3~6 个关键词
- 优先体现“实践点/工具/关键概念”，例如：`tcpdump`、`Wireshark`、`TLS`、`JVM`、`Redis`、`GC`、`Kubernetes`。

## 3. 标题风格（参考历史文章）

- 系列化标题：如《数据结构（一）——...》《数据结构（二）——...》
- 或偏讲义式标题：如《初识……》《深入理解……》《……实战》

## 4. 正文结构（建议默认结构）

建议按以下节奏组织内容（可根据主题调整，但要保持“讲义式 + 实战验证”的感觉）：

1) 背景 / 这篇要解决什么问题
2) 核心概念 / 原理（关键术语用 **加粗**）
3) 分类/对比（表格或列表）
4) 实战/示例（必须可复现：代码/命令/配置；给出验证方法）
5) 常见坑/误区/边界条件（必须有）
6) 总结（要点列表）
7) 参考资料（3~8 条，尽量权威）

## 5. 排版与表达约定

- 关键术语：使用 **加粗**
- “注：”说明：建议使用引用块或代码块承载，保持短、明确
- 代码块：可复制运行；说明输入/输出或验证方式
- 表格：用于对比（复杂度、优缺点、参数含义等）
- 图示（可选）：Mermaid/sequenceDiagram/graph（用于流程解释）

## 6. 质量门槛（发布前自检清单）

- [ ] 是否明确目标读者与先修？
- [ ] 是否至少包含 1 处可复现实验/命令/代码/配置？
- [ ] 是否明确写了“如何验证结论”？
- [ ] 是否包含“常见坑/误区”？
- [ ] 是否插入 `<!-- more -->`？
- [ ] categories 是否严格白名单且层级写法正确？
- [ ] 是否与本仓库历史文章的口吻/节奏一致？

## 7. 维护规则

- 默认以本风格文档为准；每累计 10 篇文章复盘一次并更新本文件。
