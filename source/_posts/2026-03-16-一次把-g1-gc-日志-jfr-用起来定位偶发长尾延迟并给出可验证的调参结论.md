<h1 id="0-这篇文章适合谁"><a href="#0-这篇文章适合谁" class="headerlink" title="0 这篇文章适合谁"></a>0 这篇文章适合谁</h1>

- 你有 Java 服务（Spring Boot 也好，裸 HttpServer 也行），压测下看到 <strong>P95/P99</strong>偶尔抖一下。
- 你已经听说过 “GC 会导致 STW”，但不知道怎么<strong>从证据出发</strong>定位与验证。
- 你希望能把“调 GC”从玄学变成：<strong>改一个参数 → 重新压测 → 用日志/JFR 验证</strong>。

先修：知道 JVM/GC 基本概念（Young/Old、STW），会跑命令行。

<h1 id="1-先讲清楚：我们要抓的到底是什么"><a href="#1-先讲清楚：我们要抓的到底是什么" class="headerlink" title="1 先讲清楚：我们要抓的到底是什么"></a>1 先讲清楚：我们要抓的到底是什么</h1>

这里我把几个常见名词用人话捋一遍：

- <strong>Tail latency（长尾延迟）</strong>：不是平均值慢，而是少量请求特别慢。比如 P50=40ms，P99=120ms —— 你看平均值可能还行，但用户体感会很差。
- <strong>STW（Stop-The-World）</strong>：GC 某些阶段需要暂停应用线程，暂停期间请求处理就停了，延迟自然会尖一下。
- <strong>G1 GC</strong>：JDK 默认的服务端 GC（不少版本默认就是 G1）。它目标是尽量控制暂停时间，但“尽量”不是“保证”。
- <strong>JFR（Java Flight Recorder）</strong>：JDK 自带的“黑匣子”事件采集器，能记录 GC、线程、锁、分配、异常等事件。你可以把它理解成<strong>低开销的持续采样</strong>。

这篇文章的套路是：

1) 造一个“分配压力 + 偶发大对象”的服务
2) 用简易压测脚本跑出 P99 抖动
3) 同时开 GC 日志与 JFR
4) 读证据：GC 日志看 STW 暂停分布；JFR 看 GC 事件与原因（尤其是 <strong>Humongous Allocation</strong>）
5) 给一组参数，并在同样负载下重新跑，验证“有没有变好”

<h1 id="2-可复现实验：一个会“制造垃圾”的-http-服务"><a href="#2-可复现实验：一个会“制造垃圾”的-http-服务" class="headerlink" title="2 可复现实验：一个会“制造垃圾”的 HTTP 服务"></a>2 可复现实验：一个会“制造垃圾”的 HTTP 服务</h1>

我这里用 JDK 自带的 <code>com.sun.net.httpserver.HttpServer</code> 写一个最小服务，只有两个接口：

- <code>/health</code>：健康检查
- <code>/alloc</code>：每次请求分配一段 byte[]（制造 Young GC 压力），并且“偶尔”分配一个大对象（制造 G1 的 humongous 分配场景）

下面代码可以直接粘贴保存为 <code>AllocServer.java</code>：

```java
import com.sun.net.httpserver.*;
import java.io.*;
import java.net.*;
import java.nio.charset.StandardCharsets;
import java.util.*;
import java.util.concurrent.*;

public class AllocServer {
    static final byte[] OK = "OK".getBytes(StandardCharsets.UTF_8);

    static Map<String, String> parseQuery(String q) {
        Map<String, String> m = new HashMap<>();
        if (q == null || q.isEmpty()) return m;
        for (String part : q.split("&")) {
            int i = part.indexOf('=');
            if (i <= 0) continue;
            String k = URLDecoder.decode(part.substring(0, i), StandardCharsets.UTF_8);
            String v = URLDecoder.decode(part.substring(i + 1), StandardCharsets.UTF_8);
            m.put(k, v);
        }
        return m;
    }

    static int intParam(Map<String,String> q, String k, int def) {
        try {
            if (!q.containsKey(k)) return def;
            return Integer.parseInt(q.get(k));
        } catch (Exception e) {
            return def;
        }
    }

    public static void main(String[] args) throws Exception {
        int port = Integer.parseInt(System.getProperty("port", "8080"));
        int threads = Integer.parseInt(System.getProperty("threads", "64"));
        HttpServer s = HttpServer.create(new InetSocketAddress(port), 0);
        s.setExecutor(Executors.newFixedThreadPool(threads));

        s.createContext("/health", ex -> {
            ex.getResponseHeaders().add("Content-Type", "text/plain; charset=utf-8");
            ex.sendResponseHeaders(200, OK.length);
            try (OutputStream os = ex.getResponseBody()) { os.write(OK); }
        });

        s.createContext("/alloc", ex -> {
            long t0 = System.nanoTime();
            Map<String,String> q = parseQuery(ex.getRequestURI().getRawQuery());
            int kb = intParam(q, "kb", 256);          // 每次请求分配多少 KB
            int busy = intParam(q, "busy", 0);        // 额外忙等多少 ms（模拟 CPU）
            int hum = intParam(q, "hum", 0);          // 每 N 次做一次“大对象”（触发 humongous）

            // 分配一些垃圾对象
            byte[] a = new byte[kb * 1024];
            a[0] = 1;
            a[a.length - 1] = 2;

            if (hum > 0) {
                // 用 hash 模拟“偶尔来一次”的大对象
                int h = System.identityHashCode(ex);
                if ((h & (hum - 1)) == 0) {
                    // 8MB 大对象：在 G1 里通常会走 humongous 分配
                    byte[] big = new byte[8 * 1024 * 1024];
                    big[0] = 3;
                }
            }

            if (busy > 0) {
                long end = System.nanoTime() + busy * 1_000_000L;
                long x = 0;
                while (System.nanoTime() < end) x += 7;
                if (x == 42) System.out.println("impossible");
            }

            long costUs = (System.nanoTime() - t0) / 1000;
            byte[] body = ("cost_us=" + costUs + "\n").getBytes(StandardCharsets.UTF_8);
            ex.getResponseHeaders().add("Content-Type", "text/plain; charset=utf-8");
            ex.getResponseHeaders().add("Connection", "keep-alive");
            ex.sendResponseHeaders(200, body.length);
            try (OutputStream os = ex.getResponseBody()) { os.write(body); }
        });

        System.out.println("Listening on :" + port + ", threads=" + threads);
        s.start();
    }
}
```

<h2 id="2-1-为什么这里要有-hum-这个参数"><a href="#2-1-为什么这里要有-hum-这个参数" class="headerlink" title="2.1 为什么这里要有 hum 这个参数"></a>2.1 为什么这里要有 <code>hum</code> 这个参数</h2>

因为很多“长尾抖动”的真实原因不是 Young GC 本身，而是<strong>偶发的大对象分配</strong>把 G1 拉进了更重的回收路径。

在 G1 里，如果对象大到超过某个阈值（跟 region size 有关），会按 <strong>Humongous Object</strong> 处理；你可以先把它理解成：

- 它占用的是整块 region（甚至多块）
- 分配/回收路径跟普通小对象不太一样
- 压测下更容易触发“短时间内 GC 变多、或者出现更长暂停”

<h1 id="3-压测与采集：一边跑压力，一边开证据"><a href="#3-压测与采集：一边跑压力，一边开证据" class="headerlink" title="3 压测与采集：一边跑压力，一边开证据"></a>3 压测与采集：一边跑压力，一边开证据</h1>

我这里不用 wrk/hey（很多机器上不一定装），用一个 Python 脚本做最小压测：多线程、每线程固定请求数，最后输出吞吐与 p50/p95/p99。

保存为 <code>load.py</code>：

```python
import http.client, time, threading, statistics

HOST = "127.0.0.1"
PORT = 18080
PATH = "/alloc?kb=256&busy=0&hum=16"

THREADS = 50
REQUESTS_PER_THREAD = 200
TIMEOUT = 5

lat_ms = []
err = 0
lock = threading.Lock()


def pct(xs, p):
    xs = sorted(xs)
    k = int(round((p/100.0) * (len(xs)-1)))
    return xs[k]


def worker():
    global err
    c = http.client.HTTPConnection(HOST, PORT, timeout=TIMEOUT)
    try:
        for _ in range(REQUESTS_PER_THREAD):
            t0 = time.perf_counter()
            c.request("GET", PATH, headers={"Connection":"keep-alive"})
            r = c.getresponse()
            r.read()
            if r.status != 200:
                with lock:
                    err += 1
                continue
            dt = (time.perf_counter() - t0) * 1000
            with lock:
                lat_ms.append(dt)
    except Exception:
        with lock:
            err += 1
    finally:
        try:
            c.close()
        except Exception:
            pass


start = time.time()
threads = [threading.Thread(target=worker) for _ in range(THREADS)]
for t in threads: t.start()
for t in threads: t.join()

t = time.time() - start
ok = len(lat_ms)
print(f"ok={ok} err={err} elapsed_s={t:.2f} rps={ok/t:.1f}")
if ok:
    print(f"p50_ms={pct(lat_ms,50):.2f} p95_ms={pct(lat_ms,95):.2f} p99_ms={pct(lat_ms,99):.2f} max_ms={max(lat_ms):.2f} avg_ms={statistics.mean(lat_ms):.2f}")
```

<h2 id="3-1-启动方式：默认参数（对照组）"><a href="#3-1-启动方式：默认参数（对照组）" class="headerlink" title="3.1 启动方式：默认参数（对照组）"></a>3.1 启动方式：默认参数（对照组）</h2>

这里我们开三样东西：

- G1 GC（<code>-XX:+UseG1GC</code>)
- GC 日志（<code>-Xlog:gc*</code>)
- JFR（<code>-XX:StartFlightRecording=...</code>)

> 注：JFR 是 JDK 自带的，不需要额外 agent。<strong>生产上也能开</strong>，但要注意采集时长与落盘位置。

示例命令（JDK 21）：

```bash
# 编译
javac AllocServer.java

# 启动（对照组：默认 G1 行为）
java \
  -Xms512m -Xmx512m \
  -XX:+UseG1GC \
  -Xlog:gc*:file=gc-default.log:time,uptime,level,tags \
  -XX:StartFlightRecording=filename=default.jfr,settings=profile,dumponexit=true \
  -Dport=18080 -Dthreads=64 \
  AllocServer
```

然后另开一个终端跑压测：

```bash
python3 load.py
```

我在同一台机器上得到一组样例结果（你跑出来数值可能不同，但趋势应该类似）：

- 对照组（默认 G1）：
  - ok=10000，err=0
  - rps≈1155
  - p50≈41.64ms
  - p95≈47.68ms
  - p99≈56.62ms
  - max≈121.07ms

<h2 id="3-2-如何验证：我们要把-99-分位的抖动和-gc-暂停对上号"><a href="#3-2-如何验证：我们要把-99-分位的抖动和-gc-暂停对上号" class="headerlink" title="3.2 如何验证：我们要把 99 分位的抖动和 GC 暂停对上号"></a>3.2 如何验证：我们要把 P99 的抖动和 GC 暂停对上号</h2>

验证思路很简单：

- 如果是 GC 导致的长尾，你会在 GC 日志里看到某些暂停（Pause）明显偏长；
- 在 JFR 里，你能看到 <strong>GarbageCollection 事件</strong>，并且能看到原因（cause），例如：
  - <code>G1 Evacuation Pause</code>
  - <code>G1 Humongous Allocation</code>

<h1 id="4-读证据（1）：gc-日志怎么看“暂停分布”"><a href="#4-读证据（1）：gc-日志怎么看“暂停分布”" class="headerlink" title="4 读证据（1）：GC 日志怎么看“暂停分布”"></a>4 读证据（1）：GC 日志怎么看“暂停分布”</h1>

G1 的日志一堆行，先盯住一句话：

- <code>Pause ... XXXms</code>

比如你会看到类似：

```text
... GC(64) Pause Remark 241M->16M(512M) 1.816ms
```

最粗暴的办法就是把日志里所有 Pause 的耗时拉出来做个分位数（我这里用 Python 简单解析）：

```python
import re

pauses=[]
for ln in open('gc-default.log','r',encoding='utf-8',errors='ignore'):
    m=re.search(r"\\bPause [^ ]+(?: [^ ]+)?\\s+([0-9.]+)ms\\b", ln)
    if m:
        pauses.append(float(m.group(1)))

pauses.sort()

def pct(p):
    k=int(round((p/100)*(len(pauses)-1)))
    return pauses[k]

print('count',len(pauses),'p50',pct(50),'p95',pct(95),'p99',pct(99),'max',max(pauses))
```

我这次实验的样例输出（对照组）大概是：

- Pause 分布（对照组）：
  - p50≈1.461ms
  - p95≈2.85ms
  - p99≈7.083ms
  - max≈9.382ms

看到这里你可能会吐槽：

- “GC 暂停 max 才 9ms，为啥请求 max 能到 121ms？”

这就引出 JFR 的价值：GC 的“暂停”不止你在日志里 grep 到的这几类；而且<strong>大对象导致的回收链路</strong>，经常会带来一串 GC 事件（New + Old 混在一起），总效果体现在请求侧就是长尾。

<h1 id="5-读证据（2）：jfr-怎么定位“humongous-allocation”"><a href="#5-读证据（2）：jfr-怎么定位“humongous-allocation”" class="headerlink" title="5 读证据（2）：JFR 怎么定位“Humongous Allocation”"></a>5 读证据（2）：JFR 怎么定位“Humongous Allocation”</h1>

JFR 文件可以用 JDK 自带工具读：

```bash
jfr summary default.jfr
jfr print --events jdk.GarbageCollection default.jfr | head
```

在这次实验里，我从 JFR 的 <code>jdk.GarbageCollection</code> 事件里能直接看到：

- <code>cause = "G1 Humongous Allocation"</code>

这就非常关键了：

- 说明我们制造的“偶发 8MB 大对象”确实把 G1 拉进了 humongous 分配路径；
- 这类事件往往会伴随 <strong>G1Old</strong>（并发标记/混合回收等）事件；
- 如果你的线上服务出现“偶发大对象”（比如：大 JSON、图片/文件处理、一次性聚合大 List、日志/trace 拼接大字符串），很容易复现类似症状。

我把 JFR 里 GC 事件的暂停时长做了个简单统计（单位 ms）：

- 对照组（默认）：
  - GC 事件总数 151
  - GC duration p95≈21.0ms
  - GC duration p99≈41.7ms
  - GC duration max≈65.8ms

你看，跟“日志里 Pause max 9ms”不是一个量级 —— 这是因为 JFR 的 GC 事件记录口径不同，能把某些 GC 周期的“总影响”更完整地呈现出来。

<h1 id="6-给一组可验证的-g1-参数，并重新压测"><a href="#6-给一组可验证的-g1-参数，并重新压测" class="headerlink" title="6 给一组可验证的 G1 参数，并重新压测"></a>6 给一组可验证的 G1 参数，并重新压测</h1>

先说人话版目标：

- 我们不指望让 P50 立刻变快（这更多是业务/CPU/IO 的事），
- 我们要的是：<strong>减少长暂停出现的概率</strong>，让 P99 更稳。

这组参数不是“万能答案”，它只是针对“高分配 + 偶发大对象”这种场景，给一个<strong>可复制的起点</strong>：

```bash
java \
  -Xms512m -Xmx512m \
  -XX:+UseG1GC \
  -Xlog:gc*:file=gc-tuned.log:time,uptime,level,tags \
  -XX:+UnlockExperimentalVMOptions \
  -XX:MaxGCPauseMillis=50 \
  -XX:G1NewSizePercent=20 \
  -XX:G1MaxNewSizePercent=60 \
  -XX:G1HeapRegionSize=2m \
  -XX:+ParallelRefProcEnabled \
  -XX:InitiatingHeapOccupancyPercent=30 \
  -XX:ConcGCThreads=2 \
  -XX:ParallelGCThreads=4 \
  -XX:StartFlightRecording=filename=tuned.jfr,settings=profile,dumponexit=true \
  -Dport=18080 -Dthreads=64 \
  AllocServer
```

解释一下这些参数在干啥（每个新名词都用一句话讲人话）：

- <strong>MaxGCPauseMillis</strong>：你可以把它当成“给 G1 的暂停时间 KPI”，它会尽量调年轻代大小等策略去接近这个目标。
- <strong>G1NewSizePercent / G1MaxNewSizePercent</strong>：限定年轻代占堆的比例范围，避免 G1 把年轻代搞得太小（导致 Young GC 太频繁）或太大（一次回收太重）。
- <strong>G1HeapRegionSize</strong>：G1 把堆切成很多 region；region 变大后，“大对象”更不容易被判定为 humongous（阈值跟 region 大小相关）。
- <strong>InitiatingHeapOccupancyPercent（IHOP）</strong>：老年代占用到某个比例就启动并发标记，提前做准备，避免逼到最后被迫 Full GC。
- <strong>ConcGCThreads / ParallelGCThreads</strong>：限制并发标记线程/并行回收线程数，避免 GC 抢太多 CPU（尤其是 CPU 核数不高的机器）。

然后用同样的压测参数再跑一遍，得到一组样例结果（仅供你对趋势有预期）：

- 调参组（tuned）：
  - ok=10000，err=0
  - rps≈1126
  - p50≈42.12ms
  - p95≈51.05ms
  - p99≈62.45ms
  - max≈138.66ms

看上去好像还变差了？别急，下一节我说为什么“压测结果的单次波动”不够，要用证据和多轮对照。

<h1 id="7-怎么判断调参有没有意义：不要只盯一次压测的-p99"><a href="#7-怎么判断调参有没有意义：不要只盯一次压测的-p99" class="headerlink" title="7 怎么判断调参有没有意义：不要只盯一次压测的 P99"></a>7 怎么判断调参有没有意义：不要只盯一次压测的 P99</h1>

GC 调参有个很现实的坑：

- 单轮压测的 P99 受系统噪声影响很大（CPU 抢占、页缓存、网络栈、Python 线程调度……），
- 所以正确做法是：<strong>固定负载 → 多跑几轮 → 同时看 GC 日志/JFR 的“分布变化”</strong>。

我这次实验里，从 JFR 的 GC duration 统计看到了一个比较确定的变化：

- 对照组（default JFR）：
  - GC duration p99≈41.7ms
  - GC duration max≈65.8ms
  - >=50ms 的 GC 事件：2 次
- 调参组（tuned JFR）：
  - GC duration p99≈32.1ms
  - GC duration max≈50.4ms
  - >=50ms 的 GC 事件：1 次

也就是说：在这个实验里，调参组让“最极端的 GC 事件”收敛了一点（最大值更低），但请求侧的长尾未必立刻跟着变好 —— 这很常见，因为请求长尾不只由 GC 决定。

结论更务实一点：

- <strong>如果你的线上长尾主要由 GC 引起</strong>：你会看到请求 P99 与 GC 事件/暂停时长强相关；
- <strong>如果长尾主要是别的原因</strong>（锁竞争、线程池饱和、IO 抖动）：那你调 GC 只能“修一点边角”。

<h1 id="8-常见坑-误区（踩过的人都会点头）"><a href="#8-常见坑-误区（踩过的人都会点头" class="headerlink" title="8 常见坑/误区（踩过的人都会点头）"></a>8 常见坑/误区（踩过的人都会点头）</h1>

1) <strong>只看 GC 次数，不看暂停分位数</strong>
   - Young GC 次数多不一定坏，关键是暂停是不是稳定、是不是把 tail 打爆。

2) <strong>只看 GC 日志，不开 JFR/不关联原因</strong>
   - 你会卡在“我看到 GC 了，但不知道为什么发生”。JFR 里的 <code>cause</code> 往往能直接指向问题类型。

3) <strong>误把大对象当成“G1 自己的问题”</strong>
   - 很多 humongous allocation 本质是业务创建了大对象（大数组/大字符串/大集合）。GC 只能帮你擦屁股，根因还是要回到代码。

4) <strong>拿别人的参数照抄就上生产</strong>
   - 参数必须在你的负载、你的堆大小、你的对象分配模型下验证。

5) <strong>不固定压测条件就谈结论</strong>
   - 线程数、请求模型、payload、机器规格、CPU 限制都得固定，否则对照无意义。

<h1 id="9-把它迁移到真实-spring-boot-服务：一套最小落地流程"><a href="#9-把它迁移到真实-spring-boot-服务：一套最小落地流程" class="headerlink" title="9 把它迁移到真实 Spring Boot 服务：一套最小落地流程"></a>9 把它迁移到真实 Spring Boot 服务：一套最小落地流程</h1>

你完全可以把本文流程搬到真实服务里，建议按这个顺序来：

1) 压测固定住（线程数/时长/目标 QPS/数据集）
2) 打开 GC 日志（至少 <code>-Xlog:gc*</code>）
3) 打开 JFR（先短时间，避免磁盘爆掉）
4) 从“现象”回到“证据”：
   - 请求侧：P95/P99
   - JVM 侧：GC 暂停分位数 + JFR 的 GC cause
5) 优先修业务层大对象/热点分配（这通常比调参更有效）
6) 再做 GC 参数微调，并重复验证

<h1 id="10-总结"><a href="#10-总结" class="headerlink" title="10 总结"></a>10 总结</h1>

- 长尾延迟不是靠猜的：<strong>压测 + GC 日志 + JFR</strong>，你就能把证据链搭起来。
- “偶发大对象”是长尾抖动的高频罪魁祸首之一，G1 的 <strong>Humongous Allocation</strong> 在 JFR 里能直接看到。
- GC 调参不要只盯单轮 P99，正确姿势是：<strong>多轮对照 + 看分布（p95/p99/max/次数）</strong>。
