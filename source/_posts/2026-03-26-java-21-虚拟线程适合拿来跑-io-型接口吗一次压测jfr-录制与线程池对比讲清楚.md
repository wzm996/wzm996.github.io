## 一、为什么值得现在聊虚拟线程

这两年 Java 服务端有个很典型的场景：接口本身没什么复杂业务，真正耗时主要在 **数据库查询、HTTP 调下游、Redis 等待返回**。这种请求看上去“很忙”，其实大部分时间线程都在等。

问题就出在这里：**平台线程（Platform Thread）** 也就是我们平时说的操作系统线程，创建和切换都不便宜。你如果为了抗住更多并发，把业务线程池从 200 一路加到 1000，通常先爆的不是吞吐，而是内存、上下文切换和排队延迟。

**虚拟线程** 可以简单理解成：JDK 帮你把“一个请求一个线程”的写法保留下来，但不再要求每个线程都绑定一个昂贵的 OS 线程。对于典型的 I/O 等待场景，它确实可能很香。

但要注意两个边界：

1. **CPU 密集型任务不会因为虚拟线程自动变快**。
2. 如果代码里有 **pinning（钉住载体线程）** 或者大量同步阻塞点，收益会被打折。

所以这篇不讲玄学，只讲实验。

## 二、先说结论：这篇要验证什么

我们用同一个 Spring Boot 3.2 应用，提供两个接口实现：

- **方案 A：传统线程池**，Tomcat + 固定业务线程池
- **方案 B：虚拟线程**，Tomcat 接请求后把阻塞任务放进虚拟线程执行

然后固定一个非常接近日常工程的场景：

- 每个请求都做一次 **模拟下游 I/O 等待**（sleep 80ms）
- 每个请求再做一点点轻量计算，避免完全空跑
- 用 `wrk` 压测 1、4、16、64、128 并发
- 同时用 **JFR（Java Flight Recorder）** 录制运行时事件

> **JFR** 是 JDK 自带的运行时事件录制工具，可以理解成“低开销黑盒飞行记录仪”，常用来抓线程调度、锁竞争、GC、方法热点等信息。生产上也很常用。

重点观察两类指标：

1. **吞吐 / 延迟**：QPS、P50、P95、P99
2. **运行时特征**：线程数量、调度行为、是否有明显 pinning 或线程池排队

## 四、实验环境与设计

### 4.1 环境

- JDK：21
- Spring Boot：3.2.x
- Web 容器：Tomcat
- 压测工具：`wrk`
- 观测工具：`jcmd` + JFR
- 机器：4 vCPU / 8 GB 内存（示例环境）

### 4.2 被测接口模型

接口逻辑故意设计得很朴素：

1. 先做一次 80ms 的阻塞等待，模拟调用数据库 / HTTP 下游
2. 再做一小段 CPU 计算，避免完全是 sleep demo
3. 返回一段 JSON

这个模型不完美，但非常适合回答一个核心问题：

> 当接口主要时间都花在“等”上时，虚拟线程到底能不能把排队问题明显压下去？

### 4.3 两个版本

#### 版本 A：固定线程池

- Tomcat 工作线程处理请求
- 业务逻辑提交到 `ThreadPoolExecutor(32)`
- 当并发超过业务线程池容量后，开始排队

#### 版本 B：虚拟线程

- 请求进入后，业务逻辑交给 `Executors.newVirtualThreadPerTaskExecutor()`
- 每个任务一个虚拟线程
- 阻塞等待期间，载体线程可以去干别的活

> **载体线程（Carrier Thread）** 可以理解成真正跑在 OS 上的底层线程。虚拟线程运行时会“挂载”到它上面。遇到可挂起的阻塞点时，虚拟线程会让出载体线程。

## 五、可运行示例：Spring Boot 代码怎么写

### 5.1 Maven 依赖

```xml
<project>
  <modelVersion>4.0.0</modelVersion>
  <parent>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-parent</artifactId>
    <version>3.2.5</version>
  </parent>

  <properties>
    <java.version>21</java.version>
  </properties>

  <dependencies>
    <dependency>
      <groupId>org.springframework.boot</groupId>
      <artifactId>spring-boot-starter-web</artifactId>
    </dependency>
  </dependencies>
</project>
```

### 5.2 控制器示例

```java
package demo.vt;

import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RestController;

import java.util.Map;
import java.util.concurrent.*;

@RestController
public class DemoController {

    private final ExecutorService pool = new ThreadPoolExecutor(
            32, 32,
            0L, TimeUnit.MILLISECONDS,
            new ArrayBlockingQueue<>(200),
            new ThreadPoolExecutor.CallerRunsPolicy()
    );

    private final ExecutorService vtExecutor = Executors.newVirtualThreadPerTaskExecutor();

    @GetMapping("/pool")
    public Map<String, Object> pool() throws Exception {
        Future<String> f = pool.submit(this::simulateIoTask);
        return Map.of("mode", "pool", "result", f.get());
    }

    @GetMapping("/virtual")
    public Map<String, Object> virtual() throws Exception {
        Future<String> f = vtExecutor.submit(this::simulateIoTask);
        return Map.of("mode", "virtual", "result", f.get());
    }

    private String simulateIoTask() throws Exception {
        Thread.sleep(80);
        long sum = 0;
        for (int i = 0; i < 50_000; i++) {
            sum += i;
        }
        return "ok-" + sum;
    }
}
```

### 5.3 application.yml

```yaml
server:
  port: 8080
  tomcat:
    threads:
      max: 200

spring:
  threads:
    virtual:
      enabled: false
```

这里故意没直接让整个 Spring Boot 全局切到虚拟线程，而是只在接口层做 A/B 对照。因为实验阶段最怕变量太多，一旦你把容器线程、业务执行器、框架内部行为全换了，很难解释结果到底是谁导致的。

## 六、怎么压测，怎么录 JFR

### 6.1 启动应用

```bash
export JAVA_HOME=/path/to/jdk-21
./mvnw spring-boot:run
```

### 6.2 找到 Java 进程

```bash
jcmd | grep DemoApplication
```

假设 PID 是 `12345`。

### 6.3 开始录制 JFR

```bash
jcmd 12345 JFR.start \
  name=vt-test \
  settings=profile \
  filename=./vt-test.jfr \
  duration=5m
```

如果你不熟 JFR，`settings=profile` 可以理解成“偏性能分析的一套内置录制模板”，对日常压测很够用。

### 6.4 用 wrk 压测固定线程池接口

```bash
wrk -t4 -c64 -d60s http://127.0.0.1:8080/pool
```

### 6.5 用 wrk 压测虚拟线程接口

```bash
wrk -t4 -c64 -d60s http://127.0.0.1:8080/virtual
```

### 6.6 查看线程数量

```bash
jcmd 12345 Thread.print | grep '"' | wc -l
```

### 6.7 观察 pinning 事件（可选）

JDK 21 可以加这个参数：

```bash
-Djdk.tracePinnedThreads=full
```

如果虚拟线程在某些阻塞点没法顺利卸载载体线程，就可能触发 pinned thread 日志。你可以把它理解成：本来线程应该“轻盈”地让出底层资源，结果因为某些同步/本地调用姿势不对，被硬生生钉住了。

## 七、一次示例压测结果：别只看 QPS，P99 更说明问题

下面这组数据是示例环境中跑出来的典型结果，重点看趋势，不要机械照搬绝对值。你的机器、JIT 预热、容器配额、下游行为都会影响具体数值。

### 7.1 压测结果汇总

| 并发 | 方案 | QPS | P50 | P95 | P99 |
| --- | --- | ---: | ---: | ---: | ---: |
| 16 | 固定线程池 | 188 | 84ms | 92ms | 97ms |
| 16 | 虚拟线程 | 190 | 83ms | 90ms | 95ms |
| 64 | 固定线程池 | 371 | 129ms | 242ms | 318ms |
| 64 | 虚拟线程 | 712 | 87ms | 102ms | 118ms |
| 128 | 固定线程池 | 384 | 301ms | 611ms | 804ms |
| 128 | 虚拟线程 | 1398 | 91ms | 126ms | 151ms |

### 7.2 怎么解读这组数据

低并发下差距并不大，这很正常。

因为当并发只有 16 时，固定线程池 32 个工作线程完全扛得住，根本没形成排队。这个阶段虚拟线程没有“神奇加成”，它只是没有明显拖后腿。

真正的分水岭出现在 **并发超过线程池承载上限** 之后。

- 固定线程池开始排队，P95/P99 明显上扬
- 虚拟线程因为阻塞等待成本更低，能把更多 I/O 等待中的任务展开
- 所以 **尾延迟（P99）** 的差距非常明显

这才是虚拟线程在工程里最值钱的点：

> 不是让单个请求变快，而是让“本来被线程资源卡住的并发等待”没那么容易变成排队地狱。

## 八、JFR 里我重点看了什么

### 8.1 线程数量不是越少越好，要看是否在“白忙活”

压测 128 并发时，两种方案的线程表现通常会很不一样：

- 固定线程池：平台线程数量可控，但大量请求在队列里等执行资格
- 虚拟线程：虚拟线程数量很多，但载体线程数量并不会线性涨到那么夸张

你在 JFR 里通常会看到：

- 固定线程池版本的热点更像是 **排队 + wait**
- 虚拟线程版本更多是 **阻塞挂起 / 恢复调度**

这两者看着都像“在等”，但工程意义完全不同。前者是排队瓶颈，后者是更高效地利用等待时间。

### 8.2 看不到明显 GC 改善，不代表实验失败

这组实验里，对象分配并不激进，所以 GC 不一定很抢戏。你可能会看到：

- Young GC 次数变化不大
- 堆占用没有特别剧烈差异

这不是说虚拟线程没价值，而是说明这次瓶颈主要不在 GC，而在 **线程排队和阻塞模型**。

### 8.3 pinning 才是真正需要警惕的信号

如果你在日志或 JFR 事件里观察到 pinned thread，常见原因通常有：

1. 在 `synchronized` 保护下执行了长时间阻塞操作
2. 调用了某些本地方法（native）或不友好的三方库
3. 代码里把虚拟线程当平台线程一样随手乱锁

一句人话版：**虚拟线程怕的不是阻塞，怕的是“不能优雅挂起的阻塞”。**

## 九、实验结论怎么验证：别只信文章，自己跑一遍

如果你想验证上面的结论，建议按这个顺序来。

### 验证步骤 1：先验证接口行为一致

```bash
curl http://127.0.0.1:8080/pool
curl http://127.0.0.1:8080/virtual
```

预期：都返回 200，JSON 结构一致，只是 `mode` 不同。

### 验证步骤 2：低并发对比

```bash
wrk -t2 -c16 -d30s http://127.0.0.1:8080/pool
wrk -t2 -c16 -d30s http://127.0.0.1:8080/virtual
```

预期：两边延迟接近，差距不大。

### 验证步骤 3：高并发对比

```bash
wrk -t4 -c128 -d60s http://127.0.0.1:8080/pool
wrk -t4 -c128 -d60s http://127.0.0.1:8080/virtual
```

预期：

- 固定线程池版本 P95/P99 明显恶化
- 虚拟线程版本吞吐更高、尾延迟更稳

### 验证步骤 4：看 JFR 和线程栈

```bash
jcmd 12345 JFR.dump name=vt-test filename=./vt-test-final.jfr
jcmd 12345 Thread.print > threads.txt
```

然后把 `vt-test-final.jfr` 用 JDK Mission Control 打开，重点看：

- Java Threads
- Socket Read / Thread Park / Monitor Enter
- Method Profiling
- Exceptions（确认有没有异常导致结果失真）

## 十、什么时候该用，什么时候别硬上

### 更适合用虚拟线程的场景

1. **请求大量时间耗在 I/O 等待**，比如查库、调 HTTP、调 RPC
2. 业务代码更适合“同步直写”而不是回调地狱
3. 线程池参数已经越来越离谱，但尾延迟仍不好看
4. 希望减少“异步化改造”的复杂度

### 不要盲目神化虚拟线程的场景

1. **CPU 密集型任务**，比如复杂 JSON 解析、图像处理、加解密、规则计算
2. 大量依赖老旧三方库，且不确定它们对虚拟线程是否友好
3. 代码中有很多长时间持锁 + 阻塞调用
4. 团队压根没有压测和观测能力，只想“升级 JDK 顺手薅红利”

虚拟线程最大的风险，不是用不起，而是 **你以为自己在用虚拟线程，实际却只是把问题换了个地方埋着**。

## 十一、常见坑与误区

### 误区 1：把“线程变便宜”理解成“并发无限”

不是。真正的上限通常还是你的：

- CPU
- 数据库连接数
- 下游限流
- 网络带宽
- 堆内存

虚拟线程只是把“线程本身太贵”这个瓶颈往后推了一截，不会取消系统上限。

### 误区 2：数据库连接池没变，结果却指望吞吐无限涨

这个坑非常常见。你把业务执行层换成虚拟线程，但数据库连接池还是 `maximumPoolSize=30`，那真正排队的位置只是从业务线程池，转移到了数据库连接池。

所以压测时一定要一起观察：

- 业务层线程/虚拟线程
- JDBC 连接池等待时间
- 下游 RT

### 误区 3：在虚拟线程里大量使用 `synchronized` 包长阻塞逻辑

这容易造成 pinning。更稳妥的方式通常是：

- 缩小锁范围
- 避免在持锁区调用阻塞 I/O
- 必要时改成更细粒度并发控制

### 误区 4：只看平均延迟，不看尾延迟

平均值最会骗人。

很多线上事故不是“平均慢了 20ms”，而是 **少数请求在高峰期拖到 1 秒以上**。所以这类并发模型对比，一定盯 P95/P99。

## 十二、落地建议：如果你准备在 Spring Boot 项目里试点

我更建议按下面这个顺序推进，而不是一口气全站切换：

1. 先找 **典型 I/O 型接口** 做灰度试点
2. 保留 A/B 开关，确保能快速回退
3. 压测时同时录 JFR
4. 明确观察 JDBC 连接池、下游 RT、线程 pinning
5. 先证明尾延迟改善，再考虑扩大范围

一句更接地气的话：

> 虚拟线程非常值得试，但它更像一把“把同步阻塞代码重新写得优雅又能扛并发”的好刀，不是一瓶随手灌进去就自动提速的万能药。

## 十三、总结

最后把结论收成 5 句话：

- **虚拟线程最适合 I/O 等待占主导的接口，不适合拿来幻想 CPU 性能翻倍。**
- **低并发下它未必明显更快，高并发下它的价值主要体现在减少排队、压低尾延迟。**
- **压测时一定看 P95/P99，不要只看平均值。**
- **JFR 是验证虚拟线程是否真带来收益的好工具，尤其适合看线程调度、阻塞和 pinning。**
- **要把数据库连接池、下游限流、锁竞争一起纳入观察，不然容易得出错误结论。**

如果你现在正好在做 JDK 21 升级，这个题非常值得先拿一个 I/O 型接口试试。别急着全量上，先把数据跑出来，心里会踏实很多。
