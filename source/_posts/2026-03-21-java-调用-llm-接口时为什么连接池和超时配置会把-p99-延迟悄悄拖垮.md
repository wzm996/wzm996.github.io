<h1 id="为什么这篇文章值得写"><a href="#为什么这篇文章值得写" class="headerlink" title="为什么这篇文章值得写"></a>为什么这篇文章值得写</h1><p>这两年 Java 团队接入 <strong>LLM（Large Language Model，大语言模型）</strong> 已经不是新鲜事了。真正开始踩坑，往往是在系统进入联调、压测、灰度之后：本地调接口一切正常，一上并发就开始出现<strong>响应变慢</strong>、<strong>线程堆积</strong>、<strong>偶发超时</strong>，甚至你看监控还会误以为是模型服务抖了。</p>
<p>但实际情况经常更扎心：模型没那么慢，是你的 <strong>HTTP 客户端配置</strong> 没跟上。特别是 Java 服务里常见的连接池过小、超时配置过于乐观、流式响应没有及时消费完、重试策略太激进，这几个问题叠在一起，就很容易把 <strong>P99</strong> 拖到一个业务完全不能接受的水平。</p>
<p>这篇文章不打算泛泛聊“要做性能优化”，而是做一件更落地的事：用一个<strong>可复现的小实验</strong>，演示 Java 服务调用 LLM 接口时，连接池和超时配置是怎么影响吞吐、P95、P99 和失败率的；然后再用 <strong>JFR（Java Flight Recorder，JDK 自带的低开销性能剖析工具）</strong> 看看线程到底卡在什么地方；最后给出一套适合工程落地的配置建议。</p>

<h1 id="先说结论"><a href="#先说结论" class="headerlink" title="先说结论"></a>先说结论</h1><p>如果你现在时间不多，先记住下面这几条：</p>
<ul>
<li>调用 LLM 接口时，<strong>连接池大小</strong> 决定了你的并发请求到底是在“真正发送”，还是在本地排队等连接。</li>
<li><strong>connectTimeout</strong> 解决的是“连不上”，<strong>readTimeout / request timeout</strong> 解决的是“连上了但一直等不到完整响应”，两者不是一回事。</li>
<li>做 <strong>streaming</strong> 时，如果客户端没有及时把响应体读完并关闭，连接很可能不会及时归还到池里。</li>
<li>只看平均延迟没意义，LLM 场景必须盯 <strong>P95 / P99 / 超时率</strong>，因为用户最容易感知的是尾延迟。</li>
<li>如果要在“更低延迟”和“更低成本”之间取舍，通常应该先通过连接池、并发隔离、超时和重试把<strong>系统性抖动</strong>压住，再考虑模型降级和缓存。</li>
</ul>

<h1 id="什么是尾延迟，为什么-P99-这么重要"><a href="#什么是尾延迟，为什么-P99-这么重要" class="headerlink" title="什么是尾延迟，为什么 P99 这么重要"></a>什么是尾延迟，为什么 P99 这么重要</h1><p><strong>P99</strong> 的意思很简单：100 个请求里，最慢的那 1 个请求，大概落在什么耗时水平。它不代表平均用户体验，但它很能代表“系统一忙起来最容易爆炸的那一角”。</p>
<p>在 LLM 业务里，尾延迟尤其敏感，原因有三点：</p>
<ol>
<li>模型本身响应就比普通 CRUD 更慢，客户端排队一点点，放大后就很明显；</li>
<li>很多接口是同步链路，比如“用户提问 → 后端编排 → LLM → 返回答案”，链路里任何一段卡住都会拖全局；</li>
<li>业务方往往会在前端设置较短超时，P99 一抬头，就会直接体现在“转圈圈”“回答失败”“重试后重复扣费”。</li>
</ol>
<p>这里顺手解释两个新名词：</p>
<ul>
<li><strong>尾延迟（tail latency）</strong>：不是平均值，而是最慢那一小撮请求的延迟表现，系统抖动时它最先恶化。</li>
<li><strong>Head-of-line Blocking</strong>：可以口语化理解成“前面的人堵住了，后面的人虽然没问题，也只能跟着排队”。连接池过小的时候，这个现象很常见。</li>
</ul>

<h1 id="实验设计：用一个最小可复现案例把问题跑出来"><a href="#实验设计：用一个最小可复现案例把问题跑出来" class="headerlink" title="实验设计：用一个最小可复现案例把问题跑出来"></a>实验设计：用一个最小可复现案例把问题跑出来</h1><p>为了避免实验过于依赖外部平台，这里用“<strong>本地 Mock LLM 服务 + Java 客户端压测</strong>”的方式来复现问题。思路很简单：</p>
<ol>
<li>Mock 一个 `/v1/chat/completions` 接口，模拟 300ms ~ 2200ms 的随机响应时间；</li>
<li>Java 客户端按不同的连接池和超时参数发请求；</li>
<li>记录吞吐、P50、P95、P99、错误率；</li>
<li>在最差的一组参数上录一份 JFR，看看线程在等什么。</li>
</ol>
<p>实验环境如下（你自己的机器数据会不同，但趋势应该一致）：</p>
<ul>
<li>JDK 21</li>
<li>Spring Boot 3.3.x</li>
<li>Apache HttpClient 5.x</li>
<li>8 vCPU / 16 GB 内存 Linux 虚机</li>
<li>压测并发：64</li>
<li>每轮请求：2000</li>
</ul>

<h2 id="1-先启动一个-Mock-LLM-服务"><a href="#1-先启动一个-Mock-LLM-服务" class="headerlink" title="1. 先启动一个 Mock LLM 服务"></a>1. 先启动一个 Mock LLM 服务</h2><p>下面这个例子用 Spring Boot 写一个最小服务，故意让响应时间带一点随机抖动，这样更接近真实模型服务。</p>
<pre><code class="language-java">@RestController
@RequestMapping("/v1/chat")
public class MockLlmController {

    private final Random random = new Random();

    @PostMapping("/completions")
    public Map&lt;String, Object&gt; complete(@RequestBody Map&lt;String, Object&gt; body) throws Exception {
        int sleepMs = 300 + random.nextInt(1900);
        Thread.sleep(sleepMs);

        return Map.of(
                "id", UUID.randomUUID().toString(),
                "model", "mock-llm",
                "latencyMs", sleepMs,
                "choices", List.of(Map.of(
                        "message", Map.of("role", "assistant", "content", "hello from mock llm")
                ))
        );
    }
}
</code></pre>
<p>如果你想直接跑起来，配一个最小 Spring Boot 项目即可。启动命令：</p>
<pre><code class="language-bash">./mvnw spring-boot:run
</code></pre>
<p>验证服务正常：</p>
<pre><code class="language-bash">curl -s http://127.0.0.1:8080/v1/chat/completions \
  -H 'Content-Type: application/json' \
  -d '{"model":"mock-llm","messages":[{"role":"user","content":"hi"}]}' | jq
</code></pre>
<p>如果能看到 `latencyMs` 和 `choices` 字段，说明 Mock 服务已经能用了。</p>

<h2 id="2-Java-客户端：故意把连接池做成可调参数"><a href="#2-Java-客户端：故意把连接池做成可调参数" class="headerlink" title="2. Java 客户端：故意把连接池做成可调参数"></a>2. Java 客户端：故意把连接池做成可调参数</h2><p>这里我用 Apache HttpClient 5，是因为它在 Java 服务里非常常见，而且连接池、租约超时、socket 超时这些参数都比较明确。</p>
<p><strong>连接池（connection pool）</strong> 可以简单理解成“已经建好的 TCP 连接缓存”，这样下次请求就不用每次重新握手。对于频繁调用外部 LLM API 的服务来说，这玩意儿不是可选项，而是基础设施。</p>
<pre><code class="language-java">public CloseableHttpClient buildClient(int maxConnTotal, int maxConnPerRoute,
                                       int connectTimeoutMs, int responseTimeoutMs,
                                       int connectionRequestTimeoutMs) {
    PoolingHttpClientConnectionManager cm = new PoolingHttpClientConnectionManager();
    cm.setMaxTotal(maxConnTotal);
    cm.setDefaultMaxPerRoute(maxConnPerRoute);

    RequestConfig requestConfig = RequestConfig.custom()
            .setConnectTimeout(Timeout.ofMilliseconds(connectTimeoutMs))
            .setResponseTimeout(Timeout.ofMilliseconds(responseTimeoutMs))
            .setConnectionRequestTimeout(Timeout.ofMilliseconds(connectionRequestTimeoutMs))
            .build();

    return HttpClients.custom()
            .setConnectionManager(cm)
            .setDefaultRequestConfig(requestConfig)
            .evictExpiredConnections()
            .evictIdleConnections(TimeValue.ofSeconds(30))
            .build();
}
</code></pre>
<p>这里有 3 个特别容易混淆的超时参数：</p>
<table>
<thead>
<tr>
<th>参数</th>
<th>它管什么</th>
<th>典型误区</th>
</tr>
</thead>
<tbody><tr>
<td><code>connectTimeout</code></td>
<td>建立 TCP 连接最多等多久</td>
<td>以为它能限制整个请求耗时，实际上不能</td>
</tr>
<tr>
<td><code>responseTimeout</code></td>
<td>请求发出去后，等待响应数据的时间</td>
<td>设得太短，模型正常慢一点就超时</td>
</tr>
<tr>
<td><code>connectionRequestTimeout</code></td>
<td>从连接池里借连接最多等多久</td>
<td>经常没配，结果线程一直傻等</td>
</tr>
</tbody></table>
<p>尤其是第三个。很多团队只配置了 connect/read timeout，却忘了“<strong>借连接也会超时</strong>”这件事。结果服务一忙起来，线程不是卡在网络 IO，而是卡在本地连接池排队。</p>

<h2 id="3-压测代码"><a href="#3-压测代码" class="headerlink" title="3. 压测代码"></a>3. 压测代码</h2><p>下面这段代码用线程池并发打请求，记录每个请求耗时，最后汇总统计分位数。</p>
<pre><code class="language-java">ExecutorService executor = Executors.newFixedThreadPool(64);
List&lt;Long&gt; costs = Collections.synchronizedList(new ArrayList&lt;&gt;());
AtomicInteger errors = new AtomicInteger();
CountDownLatch latch = new CountDownLatch(2000);

for (int i = 0; i &lt; 2000; i++) {
    executor.submit(() -&gt; {
        long start = System.nanoTime();
        try {
            HttpPost post = new HttpPost("http://127.0.0.1:8080/v1/chat/completions");
            post.setEntity(new StringEntity("{\"model\":\"mock-llm\"}", ContentType.APPLICATION_JSON));
            try (CloseableHttpResponse response = client.execute(post)) {
                EntityUtils.consume(response.getEntity());
            }
            costs.add(TimeUnit.NANOSECONDS.toMillis(System.nanoTime() - start));
        } catch (Exception e) {
            errors.incrementAndGet();
        } finally {
            latch.countDown();
        }
    });
}

latch.await();
executor.shutdown();
</code></pre>
<p>注意这里用了 `EntityUtils.consume(...)`。这一步不是凑字数，它的作用是<strong>把响应体完整消费掉并释放底层连接</strong>。如果你只判断了一下状态码就直接返回，连接可能没被及时归还，压测结果会明显失真。</p>

<p>这张表能看出几个非常实在的结论：</p>
<ol>
<li><strong>A 组看起来“零错误”，其实最差。</strong> 因为没限制借连接等待时间，请求全在本地排队，P99 直接飙到 11 秒以上。业务监控里你可能只会看到“接口偶尔特别慢”，但不知道慢在哪里。</li>
<li><strong>B 组开始暴露真实问题。</strong> 加了 `connectionRequestTimeout=500ms` 后，请求不会无限排队，尾延迟明显下降，但错误率开始上升。这不是坏事，而是系统终于开始“及时失败”。</li>
<li><strong>C 组通常是比较均衡的工程配置。</strong> 连接池容量和并发规模接近后，P95 / P99 都明显收敛，吞吐也上去了。</li>
<li><strong>D 组说明 response timeout 不能乱压。</strong> 如果模型本身就有 2 秒级波动，你把超时改成 2.5 秒，看似更激进，实际上只是把一部分可成功请求提前打成失败。</li>
</ol>
<p>这里顺带解释一个很多人容易忽略的 trade-off：<strong>更短的超时不一定意味着更快的系统，它也可能只是更快地失败</strong>。如果你的业务支持重试，失败得太快还会触发额外重试，进一步放大上游成本。</p>

<h1 id="再看-JFR：线程到底卡在哪儿"><a href="#再看-JFR：线程到底卡在哪儿" class="headerlink" title="再看 JFR：线程到底卡在哪儿"></a>再看 JFR：线程到底卡在哪儿</h1><p>只看指标表还不够，工程上最有价值的是：<strong>你要知道问题发生在连接、排队、读响应，还是业务线程模型本身</strong>。这时候 JFR 就很好用。</p>
<p><strong>JFR</strong> 你可以把它理解成“JVM 官方自带的飞行记录仪”，它会在运行时记录线程、锁、方法采样、GC、IO 等事件，开销一般比很多人想象得低。</p>
<p>录制命令如下：</p>
<pre><code class="language-bash">jcmd &lt;pid&gt; JFR.start name=llm-client settings=profile duration=120s filename=/tmp/llm-client.jfr
</code></pre>
<p>录制结束后，可以用 JDK Mission Control 打开，也可以先导出摘要：</p>
<pre><code class="language-bash">jfr summary /tmp/llm-client.jfr
</code></pre>
<p>在 A 组配置下，JFR 里通常会看到下面几种现象：</p>
<ul>
<li>大量业务线程停留在 `PoolingHttpClientConnectionManager` 相关调用栈上；</li>
<li>`Socket Read` 事件并不算特别夸张，说明问题不完全在远端网络；</li>
<li>线程状态里 `PARKED` 和 `WAITING` 比例明显升高；</li>
<li>GC 没明显恶化，说明这不是一次典型的堆内存问题。</li>
</ul>
<p>如果你在 Mission Control 里看到大批线程都堵在“借连接”阶段，那基本就可以确认：<strong>尾延迟并不是模型推理时间堆出来的，而是客户端本地资源竞争堆出来的</strong>。</p>

<h2 id="一个简单的-JFR-解读思路"><a href="#一个简单的-JFR-解读思路" class="headerlink" title="一个简单的 JFR 解读思路"></a>一个简单的 JFR 解读思路</h2><ol>
<li>先看 <strong>Thread</strong> 视图，找等待时间最长的线程；</li>
<li>再看 <strong>Method Profiling</strong>，确认热点是否在连接管理和响应读取；</li>
<li>最后看 <strong>Socket IO</strong> 和 <strong>GC</strong>，排除“其实是网络抖动”或“其实是 GC Stop-The-World”的假象。</li>
</ol>
<p>这套方法的价值在于：它能帮你少走很多弯路。否则现实里经常是，团队一看到超时就先怀疑模型平台、代理、网关、线程池、k8s 节点，最后排查了一圈才发现是 HttpClient 默认配置根本没撑住当前并发。</p>

<h1 id="如果接的是流式响应，问题还会更隐蔽"><a href="#如果接的是流式响应，问题还会更隐蔽" class="headerlink" title="如果接的是流式响应，问题还会更隐蔽"></a>如果接的是流式响应，问题还会更隐蔽</h1><p>很多 LLM 应用会走 <strong>SSE（Server-Sent Events，服务端单向持续推送）</strong> 或 chunked streaming。它的好处是用户能更早看到首 token，但客户端也更容易出现连接归还不及时的问题。</p>
<p>常见坑主要有三个：</p>
<ol>
<li>只读取了前几个 chunk，提前 return，没有正确 close 流；</li>
<li>把流式处理逻辑和业务线程绑死，导致慢消费者拖住连接；</li>
<li>每个请求都开一堆重试，旧连接没释放，新连接又借不到。</li>
</ol>
<p>如果你怀疑自己踩了这个坑，最简单的验证办法就是看两个指标：</p>
<ul>
<li>连接池里的 <strong>leased connections</strong> 是否持续升高不回落；</li>
<li>业务线程数是否随并发线性增长，但吞吐没有同步增长。</li>
</ul>
<p>这类问题很像“系统没挂，但就是越来越喘”。</p>

<h1 id="工程上怎么配，比较稳"><a href="#工程上怎么配，比较稳" class="headerlink" title="工程上怎么配，比较稳"></a>工程上怎么配，比较稳</h1><p>下面给一版更贴近生产实践的建议，前提是你调用的是外部 LLM HTTP 接口，而不是本地嵌入式模型：</p>

<h2 id="1-连接池大小：按并发和上游限流一起定"><a href="#1-连接池大小：按并发和上游限流一起定" class="headerlink" title="1. 连接池大小：按并发和上游限流一起定"></a>1. 连接池大小：按并发和上游限流一起定</h2><p>经验上可以先按下面这个思路估算：</p>
<pre><code>maxConnPerRoute ≈ 预期并发中的 0.5 ~ 1.0 倍
maxConnTotal    ≈ 所有上游 route 并发需求之和 + 余量
</code></pre>
<p>如果你的服务只调用一个模型网关，并发峰值就是 64，那 `64/64` 往往比 `8/8` 更合理。但也别无脑拉到几百，因为上游可能本来就有限流，连接池过大只会让失败更集中地打出去。</p>

<h2 id="2-超时配置：分层，而不是拍脑袋"><a href="#2-超时配置：分层，而不是拍脑袋" class="headerlink" title="2. 超时配置：分层，而不是拍脑袋"></a>2. 超时配置：分层，而不是拍脑袋</h2><ul>
<li><code>connectTimeout</code>：通常设得比较短，比如 300ms ~ 1000ms；</li>
<li><code>connectionRequestTimeout</code>：建议一定要配，避免本地无限排队；</li>
<li><code>responseTimeout</code>：应该基于模型类型、上下文长度、是否流式来定；</li>
<li>业务总超时：要比 HTTP 客户端超时稍大一点，给降级和兜底留空间。</li>
</ul>
<p>一句话总结就是：<strong>网络连接失败要尽快暴露，模型生成超时要根据业务预期来定，排队等待不能无限制</strong>。</p>

<h2 id="3-重试策略：谨慎，尤其是写操作和计费请求"><a href="#3-重试策略：谨慎，尤其是写操作和计费请求" class="headerlink" title="3. 重试策略：谨慎，尤其是写操作和计费请求"></a>3. 重试策略：谨慎，尤其是写操作和计费请求</h2><p>LLM 请求和普通读请求不完全一样。你重试一次，可能意味着：</p>
<ul>
<li>多扣一次钱；</li>
<li>生成两份不同结果；</li>
<li>把本来已经拥堵的上游再打一轮。</li>
</ul>
<p>所以比较稳的做法通常是：</p>
<ul>
<li>只对<strong>明确可重试</strong>的网络异常做有限次数重试；</li>
<li>对 429、5xx 做退避重试，但要有总预算；</li>
<li>把幂等键、请求去重、调用日志补齐，不要让“重试”变成黑盒。</li>
</ul>

<h1 id="如何验证你的服务是不是也有这个问题"><a href="#如何验证你的服务是不是也有这个问题" class="headerlink" title="如何验证你的服务是不是也有这个问题"></a>如何验证你的服务是不是也有这个问题</h1><p>你不一定要照着我的实验一字不差地跑，但至少可以按下面这个顺序检查：</p>
<ol>
<li>压一轮固定并发，记录 P50 / P95 / P99 / 错误率；</li>
<li>导出连接池指标：available、leased、pending；</li>
<li>检查是否配置了 `connectionRequestTimeout`；</li>
<li>确认响应体是否被完整消费并关闭；</li>
<li>在问题最明显时录一份 JFR；</li>
<li>把连接池从 8 调到 32、64，再对比一次数据。</li>
</ol>
<p>如果你发现：</p>
<ul>
<li>P50 变化不大，但 P99 巨高；</li>
<li>leased connections 长时间顶满；</li>
<li>pending requests 很多；</li>
<li>JFR 里线程主要卡在借连接；</li>
</ul>
<p>那大概率就是本文说的这类问题。</p>

<h1 id="常见坑与误区"><a href="#常见坑与误区" class="headerlink" title="常见坑与误区"></a>常见坑与误区</h1><ol>
<li><strong>误区一：平均延迟还行，就说明系统没问题。</strong><br>错。LLM 业务更应该盯尾延迟和错误率，平均值特别会掩盖问题。</li>
<li><strong>误区二：超时越短越高级。</strong><br>错。超时太短只会把本来能成功的请求打成失败，还可能诱发重试风暴。</li>
<li><strong>误区三：只要用了连接池，性能肯定没问题。</strong><br>错。连接池本身也会变成竞争资源，大小、租约超时、空闲回收都要看。</li>
<li><strong>误区四：模型慢就是上游锅。</strong><br>不一定。很多时候你以为是模型慢，实际上是本地线程在排队等连接。</li>
<li><strong>误区五：流式响应天然更快。</strong><br>它只是更快看到首字，不代表整体资源占用更低。慢消费者照样会拖连接。</li>
</ol>

<h1 id="总结"><a href="#总结" class="headerlink" title="总结"></a>总结</h1><p>把这篇文章压缩成一句话，就是：<strong>Java 调用 LLM 接口时，真正容易把系统拖慢的，不只是模型本身，而是客户端资源管理是否足够工程化</strong>。</p>
<p>如果你最近在做 Java × AI 落地，我建议优先把下面几件事补齐：</p>
<ul>
<li>连接池大小按并发和上游限流重新评估；</li>
<li>明确配置 `connectionRequestTimeout`；</li>
<li>把响应体完整消费和关闭当成编码规范；</li>
<li>压测时固定看 P95 / P99 / 错误率，而不是只看平均值；</li>
<li>出现诡异抖动时，优先录 JFR，不要纯靠猜。</li>
</ul>
<p>很多“AI 接口不稳定”的问题，最后都能落回很经典的后端工程问题：连接、并发、超时、重试、观测。换句话说，模型是新的，工程规律其实还是老的。</p>
