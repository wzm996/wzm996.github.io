<h1 id="1-先把名词讲人话：fsync、fdatasync、page-cache"><a href="#1-先把名词讲人话：fsync、fdatasync、page-cache" class="headerlink" title="1 先把名词讲人话：fsync、fdatasync、page cache"></a>1 先把名词讲人话：fsync、fdatasync、page cache</h1>

先给一句“人话版”的定义：

- <strong>page cache（页缓存）</strong>：Linux 为了让文件 I/O 看起来很快，会把写入先攒在内存里（脏页 dirty page），过一会儿再批量写到磁盘。
- <strong>fsync(fd)</strong>：告诉内核“别拖了，把这个文件相关的<strong>修改过的数据 + 必要的元数据</strong>都刷到持久化介质上，并且等到设备说‘写完了’再返回”。
- <strong>fdatasync(fd)</strong>：更偏“只要数据可靠就行”，元数据只刷“会影响后续读到数据”的那部分（比如文件大小）。

你可以把它理解成：

- write() 只是“把快递放进驿站（page cache）”，很快；
- fsync() 是“要求快递员立刻送到你家门口并签收”，慢。

> 参考：Linux man-pages 对 fsync 的描述是“flush modified in-core data … to disk device … blocks until completed”。

<h1 id="2-为什么一次-4KB-写入可能卡到-600ms：慢在链路，不在数据量"><a href="#2-为什么一次-4KB-写入可能卡到-600ms：慢在链路，不在数据量" class="headerlink" title="2 为什么一次 4KB 写入可能卡到 600ms：慢在链路，不在数据量"></a>2 为什么一次 4KB 写入可能卡到 600ms：慢在链路，不在数据量</h1>

你看到的“卡”，通常来自这些点（不一定每次都有，但只要命中一个，就够你喝一壶）：

1) <strong>文件系统语义 + journal</strong>
   - ext4 默认只对<strong>元数据</strong>做 journaling（data blocks 不保证先落盘），但 commit/journal 的写入和 flush 仍然会带来成本。
2) <strong>块层合并/调度 + 写缓存</strong>
   - fsync 要求“我已经安全了”，这往往意味着要把设备写缓存也刷掉（flush/fua）。
3) <strong>存储介质的现实</strong>
   - 即便是 SSD，“小而随机、带持久化语义的写入”也可能触发内部的 flush、GC、写放大。
4) <strong>抖动来自队列</strong>
   - 你这次 fsync 可能在等别人之前的 I/O 先落完。

所以关键认知是：

> <strong>fsync 的延迟，和你这次写了多少字节关系没你想象的大；它更像是一次“强制把队列里的东西结算掉”的动作。</strong>

<h1 id="3-可复现实验：同样写-1MB，fsync-每次刷 vs 最后刷一次"><a href="#3-可复现实验：同样写-1MB，fsync-每次刷-vs-最后刷一次" class="headerlink" title="3 可复现实验：同样写 1MB，fsync 每次刷 vs 最后刷一次"></a>3 可复现实验：同样写 1MB，fsync 每次刷 vs 最后刷一次</h1>

这节是“硬核门槛”：我们不靠嘴说，直接跑实验。

<h2 id="3-1-实验目标"><a href="#3-1-实验目标" class="headerlink" title="3.1 实验目标"></a>3.1 实验目标</h2>

对比两种写法的延迟分布：

- 模式 A：每写 4KB 就 fsync（最“安全”、通常也最慢）
- 模式 B：写完一整段再 fsync 一次（批量落盘）

<h2 id="3-2-实验代码（C-版，几十行，可复制）"><a href="#3-2-实验代码（C-版，几十行，可复制）" class="headerlink" title="3.2 实验代码（C 版，几十行，可复制）"></a>3.2 实验代码（C 版，几十行，可复制）</h2>

把下面代码保存为 <code>/tmp/fsync_bench.c</code>：

```c
#define _GNU_SOURCE
#include <errno.h>
#include <fcntl.h>
#include <inttypes.h>
#include <stdint.h>
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <sys/stat.h>
#include <time.h>
#include <unistd.h>

static uint64_t now_ns() {
  struct timespec ts;
  clock_gettime(CLOCK_MONOTONIC, &ts);
  return (uint64_t)ts.tv_sec * 1000000000ull + (uint64_t)ts.tv_nsec;
}

static void die(const char* msg) {
  perror(msg);
  exit(1);
}

int main(int argc, char** argv) {
  const char* path = "/tmp/fsync-bench.dat";
  size_t block_kb = 4;
  int blocks = 256;   // 256 * 4KB = 1MB
  int rounds = 30;
  int do_fsync_each = 1;

  for (int i=1; i<argc; i++) {
    if (!strcmp(argv[i], "--no-fsync")) do_fsync_each = 0;
  }

  const size_t block_bytes = block_kb * 1024;
  char* buf;
  if (posix_memalign((void**)&buf, 4096, block_bytes) != 0) return 1;
  memset(buf, 'A', block_bytes);

  uint64_t* lat = calloc(rounds, sizeof(uint64_t));
  if (!lat) die("calloc");

  for (int r=0; r<rounds; r++) {
    int fd = open(path, O_CREAT|O_TRUNC|O_RDWR, 0644);
    if (fd < 0) die("open");

    uint64_t t0 = now_ns();
    for (int i=0; i<blocks; i++) {
      if (write(fd, buf, block_bytes) != (ssize_t)block_bytes) die("write");
      if (do_fsync_each) {
        if (fsync(fd) != 0) die("fsync");
      }
    }

    if (!do_fsync_each) {
      if (fsync(fd) != 0) die("fsync_end");
    }

    close(fd);
    uint64_t t1 = now_ns();
    lat[r] = t1 - t0;
  }

  // 这里只打印平均值，想看 p95/p99 可以自己加排序
  double sum_ms = 0;
  for (int i=0; i<rounds; i++) sum_ms += lat[i] / 1e6;
  printf("avg_ms=%.3f\n", sum_ms / rounds);
  return 0;
}
```

编译运行：

```bash
gcc -O2 -Wall -o /tmp/fsync_bench /tmp/fsync_bench.c

# 模式 A：每次写 4KB 就 fsync
/tmp/fsync_bench

# 模式 B：最后才 fsync 一次
/tmp/fsync_bench --no-fsync
```

<h2 id="3-3-我在本文环境下的实测结果（含-p50-p95-p99）"><a href="#3-3-我在本文环境下的实测结果（含-p50-p95-p99）" class="headerlink" title="3.3 我在本文环境下的实测结果（含 p50/p95/p99）"></a>3.3 我在本文环境下的实测结果（含 p50/p95/p99）</h2>

说明：每轮写入总量 1MB（256 * 4KB），重复 30 轮；文件系统 ext4。

| 模式 | 含义 | avg (ms) | p50 (ms) | p95 (ms) | p99 (ms) |
|---|---|---:|---:|---:|---:|
| write+fsync(每 4KB) | 每次 write 后都 fsync | 613.284 | 610.573 | 658.346 | 667.841 |
| write(不刷)+最后 fsync | 只在最后 fsync 一次 | 5.617 | 5.360 | 6.167 | 7.840 |

差距大概是：<strong>~100x</strong>。

你可以把这个结果当成一个直觉锚点：

- “每次提交都要求落盘”会把吞吐压到很低；
- “批量刷盘”能显著提升吞吐，但会牺牲“崩溃时最多丢多少”的上界。

<h2 id="3-4-如何验证你的结论没跑偏"><a href="#3-4-如何验证你的结论没跑偏" class="headerlink" title="3.4 如何验证你的结论没跑偏"></a>3.4 如何验证你的结论没跑偏</h2>

你跑完实验后，建议做三件事确认“不是偶然”：

1) 把 rounds 拉大（比如 100），看 p95/p99 是否更明显。
2) 试试把 blocks 变大（比如 4096，总量 16MB）。如果“每 4KB fsync”依然慢得离谱，说明瓶颈真在 flush 语义上。
3) 在同一台机器上跑两次对比。如果两次结果差距很大，说明你的存储层抖动较强（队列/后台 GC/共享盘等）。

<h1 id="4-从应用到内核：一次-fsync-大概经历了什么"><a href="#4-从应用到内核：一次-fsync-大概经历了什么" class="headerlink" title="4 从应用到内核：一次 fsync 大概经历了什么"></a>4 从应用到内核：一次 fsync 大概经历了什么</h1>

这里不钻代码细节，只给“能解释清楚现象”的链路。

1) 你的 write() 大概率只是把数据写进 page cache，并把页标成 dirty。
2) fsync() 会触发/等待：
   - 相关脏页写回（writeback）
   - 相关元数据更新（inode/目录项等）
   - 文件系统 journal 的提交（ext4/jbd2）
   - 块层 flush（让设备写缓存也落盘）
3) 只有当设备确认写入完成，fsync 才会返回。

> ext4 的官方文档也明确：journal 记录提交点，崩溃后可以重放到最新 commit，保证元数据更新不会“卡在半路”。但默认模式下，文件数据块不一定通过 journal 保证先落盘。

<h1 id="5-常见坑：你以为安全，其实不一定；你以为慢，其实是你用法不对"><a href="#5-常见坑：你以为安全，其实不一定；你以为慢，其实是你用法不对" class="headerlink" title="5 常见坑：你以为安全，其实不一定；你以为慢，其实是你用法不对"></a>5 常见坑：你以为安全，其实不一定；你以为慢，其实是你用法不对</h1>

<h2 id="5-1-只-fsync-文件不够：目录项也可能需要-fsync"><a href="#5-1-只-fsync-文件不够：目录项也可能需要-fsync" class="headerlink" title="5.1 只 fsync 文件不够：目录项也可能需要 fsync"></a>5.1 只 fsync 文件不够：目录项也可能需要 fsync</h2>

这是很多人第一次踩的坑：

- 你创建了新文件（或 rename 做原子替换），即使 fsync 了文件本身，<strong>目录项</strong>（这个文件名到 inode 的映射）也可能还在缓存里。
- 更严谨的做法是：fsync(file) 之后，再打开目录并 fsync(dir)。

> 这也是为什么很多数据库/日志系统会非常小心地处理“落盘”的边界语义。

<h2 id="5-2-把-every-request-都-fsync：你是在用“正确方式”把系统打爆"><a href="#5-2-把-every-request-都-fsync：你是在用“正确方式”把系统打爆" class="headerlink" title="5.2 把 every request 都 fsync：你是在用“正确方式”把系统打爆"></a>5.2 把 every request 都 fsync：你是在用“正确方式”把系统打爆</h2>

如果你把 fsync 放在每个请求的关键路径上：

- 你等到的不只是你自己的 I/O；你还在帮系统把队列结算掉。
- 最终表现就是：吞吐下降 + 延迟尾部变长（p99 极难看）。

解决思路通常是：

- <strong>批量</strong>（group commit / batch）
- <strong>异步</strong>（异步提交，接受小窗口丢失风险）
- <strong>分离</strong>（把持久化从请求线程剥离，使用队列+后台 flush）

<h1 id="6-把结论映射到数据库参数：MySQL、PostgreSQL-到底在“刷”什么"><a href="#6-把结论映射到数据库参数：MySQL、PostgreSQL-到底在“刷”什么" class="headerlink" title="6 把结论映射到数据库参数：MySQL、PostgreSQL 到底在“刷”什么"></a>6 把结论映射到数据库参数：MySQL、PostgreSQL 到底在“刷”什么</h1>

如果你做业务，最关心的往往不是“fsync 的定义”，而是：

> 我把某个参数从 1 改成 2，能快多少？会丢什么？

<h2 id="6-1-MySQL-InnoDB：innodb-flush-log-at-trx-commit（典型的-1-vs-2-vs-0）"><a href="#6-1-MySQL-InnoDB：innodb-flush-log-at-trx-commit（典型的-1-vs-2-vs-0）" class="headerlink" title="6.1 MySQL/InnoDB：innodb_flush_log_at_trx_commit（典型的 1 vs 2 vs 0）"></a>6.1 MySQL/InnoDB：innodb_flush_log_at_trx_commit（典型的 1 vs 2 vs 0）</h2>

把它简单理解成“事务提交时 redo log 到底刷不刷盘”：

- <strong>1（默认，最安全）</strong>：每次提交都写 redo 并 flush 到磁盘（追求 ACID）。
- <strong>2（折中）</strong>：每次提交写 redo 到 OS cache，但刷盘由 OS 定时做（通常 1s）。
- <strong>0（最激进）</strong>：提交时不管，redo 每秒写+刷一次。

这和我们前面的实验几乎是同一个 trade-off：

- “每次提交都 fsync” → 低吞吐、高尾延迟，但崩溃恢复时最靠谱
- “每秒刷一次” → 吞吐上去，但可能丢最近 1 秒

如果你还开了 binlog（尤其是主从/审计/增量同步），还要结合 <code>sync_binlog</code> 一起看。

> 阿里云 RDS 的文档就明确把这两个参数描述为“在性能与数据安全之间平衡”，并给出：高安全场景建议都设为 1；高性能场景可做折中，但要接受丢数据风险。

<h2 id="6-2-PostgreSQL：asynchronous-commit-的思路是一模一样的"><a href="#6-2-PostgreSQL：asynchronous-commit-的思路是一模一样的" class="headerlink" title="6.2 PostgreSQL：asynchronous commit 的思路是一模一样的"></a>6.2 PostgreSQL：asynchronous commit 的思路是一模一样的</h2>

PostgreSQL 文档对“异步提交”讲得很直白：

- 同步提交：事务完成要等 WAL 刷到持久化存储。
- 异步提交：事务逻辑上完成就先返回，WAL 稍后再落盘，换吞吐，但崩溃可能丢最近一小段。

如果你读到这里，其实你已经理解了它的本质：<strong>还是那个 fsync 语义带来的成本和风险窗口。</strong>

<h1 id="7-工程建议：什么时候该坚持-fsync，什么时候该做批量/异步"><a href="#7-工程建议：什么时候该坚持-fsync，什么时候该做批量/异步" class="headerlink" title="7 工程建议：什么时候该坚持 fsync，什么时候该做批量/异步"></a>7 工程建议：什么时候该坚持 fsync，什么时候该做批量/异步</h1>

给一个我个人很实用的“决策 checklist”（偏面试八股，但能落地）：

1) 你的“正确性”是 <strong>能丢 0 条</strong>，还是 <strong>能丢 1 秒</strong>？先把这个写在需求里。
2) 你的写入是：
   - 低频但必须可靠（配置、账务、订单状态）→ 倾向同步落盘
   - 高频且可重建（缓存、指标、trace）→ 可以异步/批量
3) 你能不能引入 <strong>group commit</strong>？（把 N 次提交合并成 1 次 fsync）
4) 你有没有做“刷盘从请求线程剥离”？（队列+后台线程）
5) 你的存储是不是云盘/共享盘？（抖动和队列会更明显）

最后一句非常现实的话：

> <strong>“全都要”通常要靠系统设计，而不是靠一个参数。</strong>

<h1 id="8-总结"><a href="#8-总结" class="headerlink" title="8 总结"></a>8 总结</h1>

- write() 快，是因为 Linux 先写 page cache。
- fsync() 慢，是因为它要把“缓存里的变化”推进到“可恢复/不可丢”的边界上，并等待设备确认。
- 一次 fsync 的成本更多是“语义结算”和“队列等待”，而不是数据量。
- 数据库的很多 durability 参数，本质都是在这个 trade-off 上做选择：吞吐/延迟 vs 丢失窗口。
