<blockquote>
<p>很多 Java 服务一上来就把线程池配成“核心线程数 = CPU 核数 * 2”，听起来很专业，实际上经常是拍脑袋。线程数太小，吞吐跑不起来；线程数太大，切换开销、排队延迟、GC 压力又会反噬系统。与其靠感觉，不如用一轮小型压测，再配合 <strong>JFR</strong> 把热点抓出来，最后得到一组能解释、能复现、能验证的参数。</p>
</blockquote>

<h1 id="为什么线程池参数总是容易配错"><a href="#为什么线程池参数总是容易配错" class="headerlink" title="为什么线程池参数总是容易配错"></a>为什么线程池参数总是容易配错</h1>
<p>在日常开发里，很多同学对 <strong>线程池</strong> 的理解停留在“减少线程创建开销”和“限制并发数量”这两句话上，这当然没错，但还不够落地。真正到了线上，线程池参数通常会直接影响三个东西：</p>
<ul>
<li><strong>吞吐</strong>：单位时间能处理多少请求</li>
<li><strong>延迟</strong>：请求要等多久，尤其是 P95/P99 尾延迟</li>
<li><strong>稳定性</strong>：系统是不是会因为堆积、抖动、拒绝策略不合理而雪崩</li>
</ul>
<p><strong>P95/P99</strong> 是延迟分位数，口语化一点说，就是 95%/99% 的请求不会比这个值更慢。线上体验差，往往不是平均值太高，而是尾巴太长。</p>
<p>这篇文章不聊空泛原则，直接做一个能复现的小实验：同一段 Java 代码，在不同线程池参数下压测，观察吞吐、平均延迟、P95/P99，再用 <strong>JFR</strong> 看线程调度和热点情况，最后给出一套实用的调参思路。</p>

<h1 id="先说结论：线程池不是越大越好"><a href="#先说结论：线程池不是越大越好" class="headerlink" title="先说结论：线程池不是越大越好"></a>先说结论：线程池不是越大越好</h1>
<p>先把结论摆前面，后面用实验去证明：</p>
<blockquote>
<p>对于以 CPU 计算为主的任务，线程数通常接近 CPU 核数时更稳；对于存在明显 I/O 等待的任务，线程数可以适当大一些，但大到一定程度后，吞吐提升会变小，P99 延迟反而会明显变差。</p>
</blockquote>
<p>这里的 <strong>I/O</strong> 就是输入输出，比如网络调用、磁盘读写、数据库访问。它的特点是线程会“等”，所以可以允许更多线程并发补位。</p>
<p>另外，线程池参数不是只看 <code>corePoolSize</code> 和 <code>maximumPoolSize</code>。真正经常被忽略的是这几个：</p>
<ul>
<li><strong>workQueue</strong>：任务队列，决定你是“先排队”还是“先扩线程”</li>
<li><strong>RejectedExecutionHandler</strong>：拒绝策略，决定流量打满时系统怎么表现</li>
<li><strong>keepAliveTime</strong>：非核心线程空闲多久回收</li>
<li><strong>任务类型</strong>：CPU 密集、I/O 密集、混合型，这个决定调参方向</li>
</ul>

<h1 id="实验目标与环境"><a href="#实验目标与环境" class="headerlink" title="实验目标与环境"></a>实验目标与环境</h1>
<p>我们做一个最小可复现实验，目标很明确：</p>
<ul>
<li>同一台机器</li>
<li>同一段业务逻辑</li>
<li>只改变线程池参数</li>
<li>观察吞吐、平均延迟、P95、P99、拒绝数</li>
<li>配合 JFR 看 CPU 热点与线程状态</li>
</ul>
<p><strong>JFR</strong> 全称 <strong>Java Flight Recorder</strong>，它是 JDK 自带的低开销性能采样工具，可以抓方法热点、锁竞争、线程状态、GC 等信息。你可以把它理解成“JVM 自带的黑匣子”。</p>

<h2 id="建议环境"><a href="#建议环境" class="headerlink" title="建议环境"></a>建议环境</h2>
<ul>
<li>JDK 21</li>
<li>Linux / macOS 任意</li>
<li>4 核及以上机器更容易看出差异</li>
</ul>

<h2 id="实验代码"><a href="#实验代码" class="headerlink" title="实验代码"></a>实验代码</h2>
<p>下面这段代码模拟一种很常见的后端任务：先做一点 CPU 计算，再 sleep 一小段时间，表示等待下游响应。这样它不是纯 CPU 型，也不是纯 I/O 型，更接近真实业务。</p>

```java
import java.util.ArrayList;
import java.util.List;
import java.util.concurrent.*;
import java.util.concurrent.atomic.LongAdder;

public class ThreadPoolBenchmark {

    static class Stats {
        LongAdder success = new LongAdder();
        LongAdder rejected = new LongAdder();
        List<Long> latencyNanos = new CopyOnWriteArrayList<>();
    }

    public static void main(String[] args) throws Exception {
        int core = Integer.parseInt(System.getProperty("core", "4"));
        int max = Integer.parseInt(System.getProperty("max", "4"));
        int queue = Integer.parseInt(System.getProperty("queue", "100"));
        int taskCount = Integer.parseInt(System.getProperty("tasks", "20000"));
        int ioWaitMs = Integer.parseInt(System.getProperty("ioWaitMs", "8"));

        ThreadPoolExecutor executor = new ThreadPoolExecutor(
                core,
                max,
                60,
                TimeUnit.SECONDS,
                new ArrayBlockingQueue<>(queue),
                new ThreadPoolExecutor.AbortPolicy()
        );

        Stats stats = new Stats();
        long begin = System.nanoTime();
        CountDownLatch latch = new CountDownLatch(taskCount);

        for (int i = 0; i < taskCount; i++) {
            try {
                executor.execute(() -> {
                    long start = System.nanoTime();
                    try {
                        busyCpu();
                        Thread.sleep(ioWaitMs);
                        stats.success.increment();
                    } catch (Exception ignored) {
                    } finally {
                        stats.latencyNanos.add(System.nanoTime() - start);
                        latch.countDown();
                    }
                });
            } catch (RejectedExecutionException e) {
                stats.rejected.increment();
                latch.countDown();
            }
        }

        latch.await();
        long costNs = System.nanoTime() - begin;
        executor.shutdown();

        List<Long> sorted = new ArrayList<>(stats.latencyNanos);
        sorted.sort(Long::compare);

        double seconds = costNs / 1_000_000_000.0;
        long success = stats.success.sum();
        long rejected = stats.rejected.sum();
        double tps = success / seconds;

        System.out.printf("core=%d max=%d queue=%d tasks=%d ioWaitMs=%d%n", core, max, queue, taskCount, ioWaitMs);
        System.out.printf("success=%d rejected=%d tps=%.2f avgMs=%.2f p95Ms=%.2f p99Ms=%.2f%n",
                success,
                rejected,
                tps,
                avg(sorted),
                percentile(sorted, 95),
                percentile(sorted, 99));
    }

    static void busyCpu() {
        double x = 0;
        for (int i = 0; i < 20_000; i++) {
            x += Math.sqrt(i * 123.456);
        }
        if (x == 42) {
            System.out.println(x);
        }
    }

    static double avg(List<Long> arr) {
        if (arr.isEmpty()) return 0;
        long sum = 0;
        for (Long n : arr) sum += n;
        return sum / 1_000_000.0 / arr.size();
    }

    static double percentile(List<Long> arr, int p) {
        if (arr.isEmpty()) return 0;
        int idx = (int)Math.ceil(arr.size() * p / 100.0) - 1;
        idx = Math.max(0, Math.min(idx, arr.size() - 1));
        return arr.get(idx) / 1_000_000.0;
    }
}
```

<h1 id="怎么跑实验"><a href="#怎么跑实验" class="headerlink" title="怎么跑实验"></a>怎么跑实验</h1>
<p>先编译：</p>

```bash
javac ThreadPoolBenchmark.java
```

<p>再分别跑三组参数：</p>

```bash
# 方案 A：线程数偏小
java -XX:StartFlightRecording=filename=small.jfr,duration=60s,settings=profile \
  -Dcore=4 -Dmax=4 -Dqueue=100 -Dtasks=20000 -DioWaitMs=8 ThreadPoolBenchmark

# 方案 B：相对均衡
java -XX:StartFlightRecording=filename=balanced.jfr,duration=60s,settings=profile \
  -Dcore=8 -Dmax=8 -Dqueue=100 -Dtasks=20000 -DioWaitMs=8 ThreadPoolBenchmark

# 方案 C：线程数明显偏大
java -XX:StartFlightRecording=filename=large.jfr,duration=60s,settings=profile \
  -Dcore=32 -Dmax=32 -Dqueue=100 -Dtasks=20000 -DioWaitMs=8 ThreadPoolBenchmark
```

<p>如果你机器核数比较多，也别急着一口气把线程数拉到 100+。这类实验的意义不是“跑到最高”，而是找到转折点。</p>

<h2 id="如何查看-JFR"><a href="#如何查看-JFR" class="headerlink" title="如何查看 JFR"></a>如何查看 JFR</h2>
<p>如果你装了 JDK Mission Control，可以直接打开：</p>

```bash
jmc
```

<p>然后分别加载 <code>small.jfr</code>、<code>balanced.jfr</code>、<code>large.jfr</code>。重点看这几个视图：</p>
<ul>
<li><strong>Method Profiling</strong>：热点方法是不是都在业务代码，还是都耗在线程调度</li>
<li><strong>Threads</strong>：大量线程是不是频繁处于 Runnable/Blocked 切换</li>
<li><strong>Allocation</strong>：对象分配速率有没有明显抬升</li>
<li><strong>Garbage Collections</strong>：GC 次数和停顿有没有明显增加</li>
</ul>

<h1 id="一组典型结果（示例）"><a href="#一组典型结果（示例）" class="headerlink" title="一组典型结果（示例）"></a>一组典型结果（示例）</h1>
<p>下面这组数据是示意性的典型结果，重点看趋势，不要迷信绝对值。不同机器的数据会变，但规律通常不会差太多。</p>

| 方案 | core/max | queue | 吞吐 TPS | 平均延迟 | P95 | P99 | 拒绝数 |
|---|---:|---:|---:|---:|---:|---:|---:|
| A | 4/4 | 100 | 430 | 21ms | 35ms | 49ms | 1620 |
| B | 8/8 | 100 | 690 | 16ms | 24ms | 31ms | 210 |
| C | 32/32 | 100 | 710 | 23ms | 58ms | 97ms | 0 |

<p>这组结果很有意思：</p>
<ul>
<li>从 4 线程到 8 线程，<strong>吞吐明显提升</strong>，P95/P99 也更好，说明之前线程数确实不够。</li>
<li>从 8 线程继续拉到 32 线程，<strong>吞吐只涨了一点点</strong>，但 P99 明显恶化，这就说明收益已经接近顶了。</li>
<li>方案 C 虽然拒绝数变成了 0，看起来“全接住了”，但尾延迟变差，说明你只是把压力吞进系统内部了。</li>
</ul>
<p>这就是线程池调参里最常见的误区：<strong>以为不拒绝就是更好</strong>。实际上，不合理的“全吞”经常只是把问题从入口推到了队列和线程调度里。</p>

<h1 id="结合-JFR-看现象背后的原因"><a href="#结合-JFR-看现象背后的原因" class="headerlink" title="结合 JFR 看现象背后的原因"></a>结合 JFR 看现象背后的原因</h1>
<p>如果你打开三份 JFR，大概率会看到下面这些现象：</p>

<h2 id="方案-A：线程偏小"><a href="#方案-A：线程偏小" class="headerlink" title="方案 A：线程偏小"></a>方案 A：线程偏小</h2>
<ul>
<li>工作线程基本一直忙</li>
<li>任务队列堆积明显</li>
<li>拒绝开始出现</li>
<li>CPU 可能还没打满，但下游等待期间没有足够线程补位</li>
</ul>
<p>这说明任务里存在等待成分，线程数太保守了。</p>

<h2 id="方案-B：相对均衡"><a href="#方案-B：相对均衡" class="headerlink" title="方案 B：相对均衡"></a>方案 B：相对均衡</h2>
<ul>
<li>CPU 利用率更充分</li>
<li>队列可控</li>
<li>线程切换开销还没失控</li>
<li>GC 和对象分配速率没有明显恶化</li>
</ul>
<p>这个阶段通常就是比较舒服的工作点。</p>

<h2 id="方案-C：线程偏大"><a href="#方案-C：线程偏大" class="headerlink" title="方案 C：线程偏大"></a>方案 C：线程偏大</h2>
<ul>
<li>JFR 中线程数明显更多</li>
<li>上下文切换变频繁</li>
<li>Runnable 线程很多，但真正有价值的 CPU 工作没有同比增加</li>
<li>尾延迟拉长，局部还可能带出更多分配和 GC 压力</li>
</ul>
<p><strong>上下文切换</strong> 你可以理解成 CPU 在多个线程之间来回切换执行现场。线程一多，这种切换本身就要花成本，花多了，真正干活的时间就变少了。</p>

<h1 id="线程池参数到底该怎么配"><a href="#线程池参数到底该怎么配" class="headerlink" title="线程池参数到底该怎么配"></a>线程池参数到底该怎么配</h1>
<p>下面给一套实战里比较靠谱的思路，不保证一把命中，但比拍脑袋强很多。</p>

<h2 id="1-先判断任务类型"><a href="#1-先判断任务类型" class="headerlink" title="1. 先判断任务类型"></a>1. 先判断任务类型</h2>
<ul>
<li><strong>CPU 密集型</strong>：比如加解密、规则计算、图片处理。线程数先从 <code>CPU 核数</code> 或 <code>CPU 核数 + 1</code> 起步。</li>
<li><strong>I/O 密集型</strong>：比如查库、调 RPC、访问对象存储。线程数可以从 <code>CPU 核数 * 2</code> 到 <code>CPU 核数 * 4</code> 试起，但一定要压测验证。</li>
<li><strong>混合型</strong>：最常见。别套公式，直接分场景测。</li>
</ul>

<h2 id="2-队列别乱设成无界"><a href="#2-队列别乱设成无界" class="headerlink" title="2. 队列别乱设成无界"></a>2. 队列别乱设成无界</h2>
<p>很多项目直接上 <code>LinkedBlockingQueue</code> 默认无界，这个是典型坑。入口流量一大，任务先全进队列，短时间看起来系统很稳定，实际上只是把延迟和内存风险都往后拖。</p>
<p>我的建议是：</p>
<ul>
<li>优先用<strong>有界队列</strong></li>
<li>明确系统能接受的排队时长</li>
<li>明确打满后的策略：拒绝、降级、快速失败，而不是死扛</li>
</ul>

<h2 id="3-拒绝策略要和业务语义匹配"><a href="#3-拒绝策略要和业务语义匹配" class="headerlink" title="3. 拒绝策略要和业务语义匹配"></a>3. 拒绝策略要和业务语义匹配</h2>
<ul>
<li><strong>AbortPolicy</strong>：直接抛异常，适合必须显式感知失败的场景</li>
<li><strong>CallerRunsPolicy</strong>：调用线程自己执行，适合希望自然限流的场景</li>
<li><strong>DiscardPolicy / DiscardOldestPolicy</strong>：要慎用，除非你真的接受任务丢失</li>
</ul>
<p>很多异步链路的问题，不是线程池太小，而是拒绝以后没人监控，业务层还以为“已经提交成功”。</p>

<h2 id="4-把线程池监控打出来"><a href="#4-把线程池监控打出来" class="headerlink" title="4. 把线程池监控打出来"></a>4. 把线程池监控打出来</h2>
<p>最少要暴露这些指标：</p>
<ul>
<li><code>activeCount</code></li>
<li><code>poolSize</code></li>
<li><code>queueSize</code></li>
<li><code>completedTaskCount</code></li>
<li><code>rejectedCount</code></li>
<li>任务执行时间、排队时间</li>
</ul>
<p>如果这些指标都没有，线上调线程池基本就是闭眼开车。</p>

<h1 id="Spring-Boot-里的一个实用配置示例"><a href="#Spring-Boot-里的一个实用配置示例" class="headerlink" title="Spring Boot 里的一个实用配置示例"></a>Spring Boot 里的一个实用配置示例</h1>
<p>如果你在 Spring Boot 里用业务线程池，可以参考下面这类配置：</p>

```java
@Bean("orderQueryExecutor")
public ThreadPoolTaskExecutor orderQueryExecutor() {
    ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
    executor.setCorePoolSize(8);
    executor.setMaxPoolSize(8);
    executor.setQueueCapacity(200);
    executor.setKeepAliveSeconds(60);
    executor.setThreadNamePrefix("order-query-");
    executor.setRejectedExecutionHandler(new ThreadPoolExecutor.CallerRunsPolicy());
    executor.initialize();
    return executor;
}
```

<p>这类配置的重点不在于“8”这个数字，而在于你得知道它为什么是 8，为什么队列是 200，为什么拒绝策略选 <code>CallerRunsPolicy</code>。这些都应该能被压测和监控解释。</p>

<h1 id="怎么验证你的结论是不是站得住"><a href="#怎么验证你的结论是不是站得住" class="headerlink" title="怎么验证你的结论是不是站得住"></a>怎么验证你的结论是不是站得住</h1>
<p>不要只看一轮结果，至少按下面的方式验证：</p>
<ol>
<li>固定机器配置与 JDK 版本</li>
<li>每组参数至少跑 3 次，取中位结果</li>
<li>同时看 <strong>吞吐 + 平均延迟 + P95/P99 + 拒绝数</strong>，不要只盯 TPS</li>
<li>配合 JFR，确认瓶颈是在业务逻辑、线程切换，还是 GC</li>
<li>把实验里的任务类型，尽量替换成你自己的真实链路模型</li>
</ol>
<p>如果一套参数能做到下面三点，通常就算比较靠谱：</p>
<ul>
<li>吞吐接近平台期，不再有明显增长空间</li>
<li>P95/P99 没有因为多线程而突然恶化</li>
<li>拒绝策略符合业务预期，且可观测</li>
</ul>

<h1 id="常见坑与误区"><a href="#常见坑与误区" class="headerlink" title="常见坑与误区"></a>常见坑与误区</h1>
<ul>
<li><strong>误区一：线程越多越快</strong><br>线程不是免费资源。线程越多，栈内存、调度、上下文切换、缓存失效都会变重。</li>
<li><strong>误区二：不拒绝就是稳定</strong><br>很多时候只是把流量堆到队列里，最后变成超时风暴。</li>
<li><strong>误区三：所有业务共用一个大线程池</strong><br>高优任务、低优任务、慢任务、快任务混在一起，互相拖死，排障也很难看。</li>
<li><strong>误区四：只看平均延迟</strong><br>平均值很好看，不代表用户体验好。尾延迟才是真正容易把服务口碑打坏的东西。</li>
<li><strong>误区五：只调线程池，不看下游</strong><br>有些场景根本不是线程池问题，而是数据库连接池、远程依赖 RT、限流阈值出了问题。</li>
</ul>

<h1 id="总结"><a href="#总结" class="headerlink" title="总结"></a>总结</h1>
<ul>
<li>线程池参数不能靠“经验数字”直接拍，至少要结合任务类型、压测结果和运行时画像来定。</li>
<li>对于混合型任务，线程数增加通常先带来吞吐提升，过了拐点之后，收益会迅速变小，P99 反而恶化。</li>
<li>JFR 是非常适合做这类问题定位的工具，成本低、信息够全，值得日常用起来。</li>
<li>真正靠谱的线程池配置，应该能回答三件事：为什么这么配、打满会怎样、怎么监控验证。</li>
</ul>
<p>如果你愿意再往前走一步，我很建议把线程池调优和 <strong>压测脚本</strong>、<strong>JFR 采样命令</strong>、<strong>Prometheus 指标</strong> 一起沉淀成团队模板。这样下次再遇到“线程池该配多少”这种问题，就不是群里拍脑袋，而是拿数据说话了。</p>
