## 0. 这篇写给谁？先修是什么？

- 目标读者：Java 后端、SRE、做性能/稳定性的人（尤其是碰到“P95 飙高但不知道从哪下手”的场景）。
- 先修：会基本的 Linux / Docker / curl；知道什么是 P50/P95/P99；了解一点 JVM 进程的概念就够了。

顺便把几个名词先说人话：

- **JFR（Java Flight Recorder）**：JDK 自带的事件录制器，能把 JVM 里发生的“关键事”（CPU 采样、GC、线程、锁、异常、类加载……）录到文件里，之后用工具回放分析。
- **JMC（Java Mission Control）**：打开/分析 .jfr 的 GUI 工具（Mac/Windows/Linux 都有），里面能直接看火焰图/热点/线程等视图。
- **jcmd**：给 JVM 发送诊断命令的工具（同机同用户），用它可以 start/stop/dump JFR。

## 1. 我们用一个最小可复现实验：让延迟“炸”一次

为了让你在自己机器上 10 分钟复现，我这里用一个超小的 HTTP 服务：

- `/fib?n=35`：故意用一个很慢的递归 fib，当作“热点 CPU 逻辑”。
- 加一个开关 `-Dopt=true`：切成迭代实现，模拟“优化后”。

注意：这不是为了讲 fib，是为了把“采样 + 定位热点 + 验证优化”的流程跑通。

### 1.1 准备代码（App.java）

把下面文件保存到一个空目录，比如 `jfr-demo/App.java`：

```java
import com.sun.net.httpserver.HttpExchange;
import com.sun.net.httpserver.HttpHandler;
import com.sun.net.httpserver.HttpServer;

import java.io.IOException;
import java.io.OutputStream;
import java.net.InetSocketAddress;
import java.net.URI;
import java.nio.charset.StandardCharsets;
import java.time.Instant;
import java.util.HashMap;
import java.util.Map;

public class App {
    // Toggle with -Dopt=true
    static final boolean OPT = Boolean.getBoolean("opt");

    public static void main(String[] args) throws Exception {
        int port = Integer.parseInt(System.getProperty("port", "8080"));
        HttpServer server = HttpServer.create(new InetSocketAddress("0.0.0.0", port), 0);
        server.createContext("/fib", new FibHandler());
        server.createContext("/health", e -> {
            byte[] b = "OK".getBytes(StandardCharsets.UTF_8);
            e.sendResponseHeaders(200, b.length);
            try (OutputStream os = e.getResponseBody()) { os.write(b); }
        });
        server.setExecutor(java.util.concurrent.Executors.newFixedThreadPool(
                Integer.parseInt(System.getProperty("threads", "8"))
        ));

        System.out.println("started at " + Instant.now() + ", port=" + port + ", opt=" + OPT);
        server.start();
    }

    static class FibHandler implements HttpHandler {
        @Override
        public void handle(HttpExchange exchange) throws IOException {
            URI uri = exchange.getRequestURI();
            Map<String,String> q = parseQuery(uri.getRawQuery());
            int n = Integer.parseInt(q.getOrDefault("n", "35"));

            long value = fib(n);
            String body = "opt=" + OPT + " n=" + n + " value=" + value + "\n";
            byte[] b = body.getBytes(StandardCharsets.UTF_8);

            exchange.getResponseHeaders().add("Content-Type", "text/plain; charset=utf-8");
            exchange.sendResponseHeaders(200, b.length);
            try (OutputStream os = exchange.getResponseBody()) {
                os.write(b);
            }
        }

        static Map<String,String> parseQuery(String raw) {
            Map<String,String> m = new HashMap<>();
            if (raw == null || raw.isEmpty()) return m;
            for (String part : raw.split("&")) {
                int i = part.indexOf('=');
                if (i <= 0) continue;
                String k = java.net.URLDecoder.decode(part.substring(0, i), StandardCharsets.UTF_8);
                String v = java.net.URLDecoder.decode(part.substring(i + 1), StandardCharsets.UTF_8);
                m.put(k, v);
            }
            return m;
        }
    }

    static long fib(int n) {
        if (OPT) return fibIterative(n);
        return fibRecursive(n);
    }

    static long fibRecursive(int n) {
        if (n <= 1) return n;
        return fibRecursive(n - 1) + fibRecursive(n - 2);
    }

    static long fibIterative(int n) {
        if (n <= 1) return n;
        long a = 0, b = 1;
        for (int i = 2; i <= n; i++) {
            long c = a + b;
            a = b;
            b = c;
        }
        return b;
    }
}
```

### 1.2 用 Docker 跑 JDK 21（不用在宿主机装 Java）

在同目录创建 `Dockerfile`：

```dockerfile
FROM eclipse-temurin:21-jdk
WORKDIR /app
COPY App.java /app/App.java
RUN javac App.java
EXPOSE 8080
# 解锁/增强采样（可选，但建议开）：
# -XX:+DebugNonSafepoints：让采样更“贴近真实热点”，尤其是你后面要用 async-profiler 一类工具时。
CMD ["java","-XX:+UnlockDiagnosticVMOptions","-XX:+DebugNonSafepoints","-Dport=8080","App"]
```

构建并启动（先跑“慢版本”，opt=false）：

```bash
docker build -t jfr-demo:1 .

docker rm -f jfr-demo >/dev/null 2>&1 || true
docker run -d --name jfr-demo -p 18080:8080 -e JAVA_TOOL_OPTIONS='-Dopt=false' jfr-demo:1

curl -s http://127.0.0.1:18080/health && echo
curl -s 'http://127.0.0.1:18080/fib?n=35'
```

如果你看到 `OK` 和 fib 的输出，就说明服务起来了。

## 2. 先用“数字”把问题钉住：P95/P99 到底有多离谱？

我这里用 Python 写了一个很土但很稳定的压测脚本（不依赖 wrk/hey/vegeta）。

保存为 `bench.py`：

```python
import time, statistics, urllib.request

def bench(url, n=200):
    lat=[]
    for _ in range(n):
        t0=time.perf_counter()
        with urllib.request.urlopen(url, timeout=5) as r:
            r.read()
        lat.append((time.perf_counter()-t0)*1000)
    lat.sort()
    def pct(p):
        idx=int(round((p/100)*(len(lat)-1)))
        return lat[idx]
    return {
        'n': n,
        'p50': pct(50),
        'p95': pct(95),
        'p99': pct(99),
        'mean': statistics.mean(lat)
    }

print(bench('http://127.0.0.1:18080/fib?n=35', n=200))
```

在我的复现环境里（同机回环，200 次请求）：

- `opt=false`：P50 ≈ **40.6ms**，P95 ≈ **68.9ms**，P99 ≈ **70.8ms**
- `opt=true`：P50 ≈ **1.5ms**，P95 ≈ **4.3ms**，P99 ≈ **6.6ms**

再跑一个并发 16 的版本（更像“线上被打”的感觉），保存为 `bench_concurrency.py`：

```python
import argparse, concurrent.futures, time, statistics, urllib.request

def one(url):
    t0=time.perf_counter()
    with urllib.request.urlopen(url, timeout=5) as r:
        r.read()
    return (time.perf_counter()-t0)*1000

def run(url, total, concurrency):
    lat=[]
    start=time.perf_counter()
    with concurrent.futures.ThreadPoolExecutor(max_workers=concurrency) as ex:
        futs=[ex.submit(one, url) for _ in range(total)]
        for f in concurrent.futures.as_completed(futs):
            lat.append(f.result())
    dur=time.perf_counter()-start
    lat.sort()
    def pct(p):
        idx=int(round((p/100)*(len(lat)-1)))
        return lat[idx]
    rps= total/dur
    return {
        'total': total,
        'concurrency': concurrency,
        'rps': rps,
        'p50': pct(50),
        'p95': pct(95),
        'p99': pct(99),
        'mean': statistics.mean(lat),
        'sec': dur,
    }

if __name__=='__main__':
    ap=argparse.ArgumentParser()
    ap.add_argument('--url', required=True)
    ap.add_argument('--total', type=int, default=400)
    ap.add_argument('--c', type=int, default=16)
    args=ap.parse_args()
    print(run(args.url, args.total, args.c))
```

我跑到的数据（total=400, c=16）：

| 版本 | RPS | P50 | P95 | P99 |
|---|---:|---:|---:|---:|
| opt=false（递归） | ~43.5 | ~358ms | ~444ms | ~580ms |
| opt=true（迭代） | ~977.8 | ~10.7ms | ~35.7ms | ~98.7ms |

看到这张表你就能很确定：这不是“偶发抖动”，是确定性的大热点导致的系统性延迟。

到这里，你已经完成了性能排障最重要的第一步：**用指标把问题量化**。

接下来才轮到“证据链”：到底谁在烧 CPU？

## 3. 开始录 JFR：不猜，直接抓现行

### 3.1 找到 JVM PID

在容器里跑：

```bash
docker exec jfr-demo jcmd
```

会看到类似：

```text
1 App
37 jdk.jcmd/sun.tools.jcmd.JCmd
```

这里 `1` 就是我们的 Java 进程 PID。

### 3.2 开始/停止录制

> 经验：排查 CPU/延迟时，优先用 `settings=profile`，它是“通用低开销”的采样配置。

```bash
PID=$(docker exec jfr-demo jcmd | awk '/App/{print $1; exit}')

docker exec jfr-demo jcmd $PID JFR.start \
  name=profile settings=profile filename=/tmp/baseline.jfr

# 这段时间你去压测/复现问题（比如跑 10~30 秒）
python3 bench_concurrency.py --url 'http://127.0.0.1:18080/fib?n=35' --total 400 --c 16

docker exec jfr-demo jcmd $PID JFR.stop name=profile

docker cp jfr-demo:/tmp/baseline.jfr ./baseline.jfr
```

验证点：你应该能在当前目录看到 `baseline.jfr`。

## 4. 分析 JFR：CPU 热点通常 30 秒内就能定位

你有两种方式：

- GUI：用 **JMC** 打开 `baseline.jfr`，点开 *Method Profiling / Flame Graph / Hot Methods*。
- CLI：在没有 GUI 的机器上，用 `jfr print` 看事件。

这里我演示 CLI 方式（因为它最“可复现”）。

### 4.1 直接看 ExecutionSample 的堆栈

```bash
# 需要 JDK 11+（自带 jfr 工具）
jfr print --events jdk.ExecutionSample --stack-depth 3 baseline.jfr | head -n 60
```

你会看到类似这样的片段（重点看 stackTrace）：

```text
jdk.ExecutionSample {
  ...
  stackTrace = [
    App.fibRecursive(int)
    App.fibRecursive(int)
    App.fibRecursive(int)
    ...
  ]
}
```

这就是“抓现行”。

- 线程状态是 `STATE_RUNNABLE`：说明线程在跑 CPU（不是在等锁、等 IO）。
- 栈顶是 `App.fibRecursive(int)`：热点方法已经指到你脸上了。

### 4.2 一个更实用的视角：统计 Top 栈帧（粗暴但好用）

很多时候你不想看一堆事件，想要“Top N 热点方法”。

用下面的 awk 统计一下每条采样里出现的栈帧（只统计前几层就够用了）：

```bash
jfr print --events jdk.ExecutionSample --stack-depth 5 baseline.jfr \
  | awk '
    /stackTrace = \[/{inStack=1; next}
    inStack && /\]/{inStack=0; next}
    inStack {
      gsub(/^\s+/,"",$0);
      sub(/ line: .*/,"",$0);
      sub(/,$/,"",$0);
      if(length($0)>0 && $0!="...") { c[$0]++ }
    }
    END{ for(k in c) printf("%8d  %s\n", c[k], k) }
  ' | sort -nr | head -n 10
```

我的一次结果（截取关键行）：

```text
2925  App.fibRecursive(int)
```

一句话结论：**CPU 主要烧在 fibRecursive 上**。

到这一步，你其实已经完成了绝大部分“定位”工作：

- 指标：P95/P99 确实很高
- 证据：采样栈把热点方法指向了具体函数

剩下的就是“修”和“验证”。

## 5. 做一次“可控”的优化，并用同样的方法验收

我们这里的优化非常简单：把递归 fib 改成迭代版（减少指数级重复计算）。

启动优化版本：

```bash
docker rm -f jfr-demo >/dev/null 2>&1 || true

docker run -d --name jfr-demo -p 18080:8080 \
  -e JAVA_TOOL_OPTIONS='-Dopt=true' jfr-demo:1
```

重复压测（同参数、同请求量、同并发）：

```bash
python3 bench_concurrency.py --url 'http://127.0.0.1:18080/fib?n=35' --total 400 --c 16
```

然后再录一份 JFR（流程完全一样）：

```bash
PID=$(docker exec jfr-demo jcmd | awk '/App/{print $1; exit}')

docker exec jfr-demo jcmd $PID JFR.start \
  name=profile settings=profile filename=/tmp/opt.jfr

python3 bench_concurrency.py --url 'http://127.0.0.1:18080/fib?n=35' --total 400 --c 16

docker exec jfr-demo jcmd $PID JFR.stop name=profile

docker cp jfr-demo:/tmp/opt.jfr ./opt.jfr
```

对比分析：你会发现 `opt.jfr` 里已经基本看不到 `App.fibRecursive` 作为热点了。

这就是一个完整闭环：

1) 用指标量化问题
2) 用 JFR 抓证据定位热点
3) 进行针对性优化
4) 用同样的压测与 JFR 方式验收

## 6. 常见坑（真线上更容易踩）

### 6.1 “我 jcmd 不进 JVM / 提示权限不够”

`jcmd` 的要求是：**同机 + 同用户（有效 UID/GID）**。

- 你在 k8s 里 exec 进去的用户和 Java 进程不是一个用户时，常见会失败。
- 解决思路：用相同用户执行，或者在容器里把诊断工具打进去并保证权限一致。

### 6.2 “录了 JFR 但啥都看不到 / 没热点”

常见原因：

- 你没在“问题发生的窗口”录制（比如录的时候没压测/没请求）。
- 你录得太短、样本太少。
- 你用的事件不对：CPU 排查优先看 `jdk.ExecutionSample`（profile 设置里默认会开）。

经验：先录 20~60 秒，确保复现窗口覆盖进去。

### 6.3 “线上能不能一直开着 JFR？”

很多团队的做法是：

- 常驻开一个低开销 recording（比如 `settings=profile` + 合理 maxsize/maxage）。
- 出问题时 dump 最近 N 分钟的数据。

这样相当于 JVM 自带一个“轻量黑匣子”。

### 6.4 “我有 async-profiler / arthas，还需要 JFR 吗？”

我的观点（偏工程落地）：

- **JFR 是通用底座**：它事件面很广（线程/锁/GC/类加载/异常），而且基本 JVM 原生支持。
- async-profiler 更强在“火焰图体验”和某些采样模式（比如 perf_event），但它更像是“你确定要深挖时的更锋利的刀”。

实际排障经常是：先 JFR 快速定方向，再决定要不要上更重的工具。

## 7. 总结（把这套流程带走）

- 先用 **压测/回放** 把 P95/P99 量化，别靠感觉。
- 再用 **JFR** 把热点抓出来：`jdk.ExecutionSample` 很好用。
- 最后做针对性优化，并用同样的压测 + JFR 验收，形成闭环。

如果你愿意把这套流程固化成 SOP：

- 线上常驻 profile recording + 发生抖动时 dump
- 统一压测脚本与对比表
- PR 里强制要求“证据链”（指标 + 录制文件 + 结论）

性能优化就会从“玄学”变成“工程”。
